Guest → Account Upgrade — Technical Design

How an anonymous user becomes a permanent Google/Apple account with their local-first data intact, using Firebase account linking to preserve the UID and the existing sync engine to push that data to the account. Gated behind a Feature Lab flag. Companion to the PRD.

1. The Key Insight

Every local record is stored under EntityScope.userRoot(uid) → /users/{uid}/…. The UID is the join key between the local SQLDelight store and Firestore. Two facts make the upgrade clean:

  • Firebase account linking preserves the UID. FirebaseUser.linkWithCredential(...) attaches a permanent credential (Google/Apple) to the same anonymous user. The UID does not change, so every local row stays valid with zero migration.
  • The sync engine already does the syncing. SyncEngine idles while user == null || user.isAnonymous. The instant the user becomes non-anonymous (same UID), it wakes, hydrates, and PushWorker drains the local dirty=1 rows to Firestore. Since guests never synced, all their rows are dirty and get pushed.
What we must NOT do The login screen's "Sign in with Google" calls signInWithCredential, which mints a new UID and orphans all guest data. The upgrade path must use linkWithCredential, never signInWithCredential, on the clean path.

2. Architecture Overview

GUEST (anonymous uid = A) PERMANENT (uid = A, linked) ┌─────────────────────────┐ ┌─────────────────────────┐ │ SQLDelight entity store │ │ SQLDelight entity store │ │ scope = users/A/… │ link(cred) │ scope = users/A/… │ │ every row dirty=1 │ ───────────▶ │ (unchanged) │ │ SyncEngine: IDLE │ same UID │ SyncEngine: ACTIVE │ └─────────────────────────┘ └───────────┬─────────────┘ │ push dirty rows ▼ Firestore /users/A/… (synced) COLLISION (cred already belongs to uid = B): link(cred) → FirebaseAuthUserCollisionException → confirm merge → signInWithCredential(cred) [now uid = B] → reassign local scope A ⟶ B (mark dirty, reset cursors) → SyncEngine hydrates B's cloud set + pushes reassigned rows (union)

3. Auth Layer Changes (core:auth)

Add an upgrade-oriented API to AuthManager. The primary provider is platform-determined: Google on Android, Apple on iOS. The result is a sealed type so the ViewModel can branch on link-success vs collision vs cancel vs error.

// commonMain — AuthManager.kt (additions)
sealed interface AccountUpgradeResult {
  data class Linked(val user: FirebaseUser) : AccountUpgradeResult      // clean: same uid, now permanent
  data class CredentialInUse(val credential: AuthCredential) : AccountUpgradeResult // collision → offer merge
  data object Cancelled : AccountUpgradeResult                          // user dismissed provider sheet
  data class Failed(val message: String) : AccountUpgradeResult
}

interface AuthManager {
  // …existing…
  /** Links the platform's primary provider (Google on Android, Apple on iOS) to the
      current anonymous user, preserving the UID. */
  suspend fun upgradeAnonymousAccount(): AccountUpgradeResult

  /** Merge path: sign in to the existing account that owns [credential]. Returns the new
      permanent user (different UID) or Failed. Caller reassigns local data afterward. */
  suspend fun signInToExistingAccount(credential: AuthCredential): AccountUpgradeResult
}

Android (androidMain)

Reuse the existing Credential Manager + Google ID-token plumbing (already present for signInWithGoogle), but call currentUser.linkWithCredential(googleCredential) instead of signInWithCredential. Catch FirebaseAuthUserCollisionException (GitLive surfaces this) and return CredentialInUse(googleCredential).

iOS (iosMain)

Implement Sign in with Apple via ASAuthorizationController with a hashed nonce, then build a GitLive OAuthProvider.credential("apple.com", idToken, rawNonce) and linkWithCredential. This also closes the current gap where iOS has no real provider (signInWithGoogle returns null on iOS). Apple is required by App Store policy once Google is offered.

App Store compliance Offering Google sign-in on iOS would require Sign in with Apple alongside it. Splitting providers by platform (Google→Android, Apple→iOS) keeps each store happy and matches each platform's native identity.

4. Clean Link Path

  1. User taps the upgrade CTA in Settings → UpgradeViewModel.startUpgrade() calls authManager.upgradeAnonymousAccount().
  2. Provider sheet returns an ID token; linkWithCredential succeeds → Linked(user). UID unchanged; user.isAnonymous is now false.
  3. SyncEngine's auth-state collector sees a non-anonymous user (and cloudSyncEnabled) → leaves idle, runs HydrationRunner (cloud is empty for a first-time UID, so nothing to pull) and PushWorker drains the dirty rows.
  4. UI shows progress, then a single "Syncing complete" state (no record count). The Settings profile card flips from guest to the real identity.

No storage migration runs on this path. The only state that changed is the Firebase user's provider set and isAnonymous flag.

5. Merge Path (Collision)

When the credential already belongs to account B, linking throws a collision. We cannot link (the credential is taken), so we sign in to B and bring the guest's local data along by re-scoping it from A to B.

  1. upgradeAnonymousAccount() returns CredentialInUse(cred). The VM shows the merge confirmation dialog (PRD §6).
  2. On confirm, capture the guest UID A = authManager.getCurrentUser().uid before switching, then call signInToExistingAccount(cred) → signed in as B. (The anonymous user A becomes orphaned in Firebase Auth; its local rows still exist, keyed by A.)
  3. Run LocalAccountMigrator.reassign(fromUid = A, toUid = B) (see §6) — re-scope local entities & blobs to B, mark them dirty=1, clear sync cursors so hydration re-runs for B.
  4. SyncEngine (now user B, non-anonymous) hydrates B's existing cloud set and pushes the reassigned dirty rows. The result is an additive union of both record sets.
  5. Success state: "Syncing complete." (Same copy as the clean path; the user is not shown a record count or a merge-specific message.)
Why re-scope instead of a server merge? The local-first store and sync engine already do the heavy lifting. Re-keying the local rows to B and marking them dirty turns "merge" into "a normal push," reusing the proven conflict model (last-writer-wins on Firestore server timestamp; dirty rows are immune from remote overwrite). No new server code, no temp staging area.

6. Local Data Re-keying (core:storage)

The merge needs one new storage operation. The store already exposes DatabaseIntegrityChecker.wipeDataForUser(uid); add a sibling that moves rather than wipes:

// core:storage — runs in a single SQLDelight transaction
interface LocalAccountMigrator {
  /** Re-scope every local row owned by [fromUid] to [toUid], mark entities dirty so the
      sync engine re-uploads them under the new account, and reset sync cursors so [toUid]
      hydrates its existing cloud set. Idempotent and safe to re-run. */
  suspend fun reassign(fromUid: String, toUid: String)
}
  • entity table: UPDATE … SET scope = users/{toUid}/…, dirty = 1 WHERE scope = users/{fromUid}/…
  • sync_cursor table: delete cursors for both UIDs so HydrationRunner re-runs for toUid.
  • blob_object table: re-scope blob ownership to toUid and reset remote state so BlobUploadDriver re-uploads (it also gates on non-anonymous).

Because reassignment is a single idempotent transaction keyed on scope, it is resumable: if it ran partially, re-running completes it; if it fully ran, re-running is a no-op (no rows match fromUid anymore).

7. Sync Integration

No change to the sync engine's logic; the upgrade simply flips the gating condition. Relevant existing behavior we rely on:

MechanismRole in upgrade
Auth-state gating (user.isAnonymous)Linking flips this to false → engine activates for the same/!new UID.
PushWorker (drains dirty=1)Uploads all guest records (clean path) or reassigned records (merge path).
HydrationRunner / PullListenerClean path: nothing to pull. Merge path: pulls account B's existing set.
Last-writer-wins + dirty immunityOn merge, reassigned local rows (dirty) are never clobbered by the remote pull before they upload.
cloudSyncEnabled master toggleIf a user has cloud sync off, upgrading still links the account; syncing begins when they enable sync. Surfaced in Backup & Sync settings.

8. UI (feature:settings + upgrade flow)

Entry point is the Settings profile area only, and is shown only when the Feature Lab flag is on and the current user is anonymous. The UserProfileCard gains a guest variant; a new upgrade flow (bottom sheet or dedicated screen) owns the steps. There is no benefits/promo screen: the CTA leads straight into the provider sign-in.

// UpgradeUiState — drives the upgrade sheet
sealed interface UpgradeUiState {
  data object Idle : UpgradeUiState              // CTA visible, sheet closed
  data object Authenticating : UpgradeUiState    // provider sheet / linking
  data class ConfirmMerge(val credential: AuthCredential, val email: String) : UpgradeUiState
  data object Working : UpgradeUiState           // pushing / reassign+sync
  data object Success : UpgradeUiState           // renders "Syncing complete" — no count, no merge-specific copy
  data class Error(val message: String) : UpgradeUiState
}

Visual language follows DESIGN.md: aviation-blue primary action, a single committed primary button per step, and a calm, definite Success state ("Syncing complete") with no record count. ConfirmMerge carries the existing account's email only to label the merge dialog, not to display a count. Provider button label and icon switch per platform (Google / Apple). The merge confirmation is an AlertDialog (a genuine destructive-adjacent decision), consistent with the logout-warning pattern already shipped.

9. Error Handling & Recovery

FailureHandling
Provider sheet cancelledCancelled → back to Idle, no change.
No networkFailed("needs internet"); retry available; local data untouched.
Collision but user cancels mergeDo not call signInToExistingAccount; remain guest A.
App killed after signInToExistingAccount but before reassign completesOn launch, if local rows exist under a UID ≠ currentUser.uid and the current user is non-anonymous, run reassign(orphanUid, currentUid). Idempotent (§6).
Push fails after a successful linkAccount is linked (upgrade done); rows stay dirty and upload later via normal sync. No data loss.
Launch-time reconciliation A startup check ("is there local data owned by a UID other than the signed-in non-anonymous user?") makes the merge crash-safe and also cleans up any future edge where the auth UID and local scope diverge.

10. Edge Cases

11. Security & Privacy

  • Apple Sign In uses a SHA-256-hashed nonce to prevent replay; raw nonce kept only for the credential exchange.
  • No new PII is stored beyond what the provider returns (email, display name) and what already syncs.
  • The orphaned anonymous Firebase user (merge path) holds no cloud data (guests never synced) and is left to Firebase's anonymous-account lifecycle.

12. Testing Plan

  • Unit: AccountUpgradeResult branching; LocalAccountMigrator.reassign re-scopes + marks dirty + clears cursors; idempotency (run twice).
  • Integration: clean link → all dirty rows pushed; merge → union of two sets; interrupted-merge recovery on simulated restart.
  • Manual matrix: Android/Google + iOS/Apple × {new account, existing account, cancel, offline, Hide-My-Email}.
  • Guardrail: assert zero record loss across every path; assert no UID-scope divergence after reconciliation.

13. Rollout

  • Gated behind a Feature Lab flag (FeatureLabManager), off by default; the Settings affordance and the whole flow are hidden until the flag is on, mirroring how other experimental surfaces ship.
  • Land core:auth + core:storage + sync wiring first (no UI), then the Settings UI behind the flag.
  • Watch conversion, collision rate, and post-merge duplicate rate (PRD §4) to decide on dedupe fast-follow and extra entry points.

14. Open Questions

  • Should reassign live on the existing DatabaseIntegrityChecker or a new dedicated type? (Leaning new type for single responsibility.)
  • iOS Apple Sign In: implement natively per-screen, or via a shared expect/actual entry to keep the Compose layer platform-agnostic?
