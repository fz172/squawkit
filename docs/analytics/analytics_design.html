Product Analytics — Technical Design

Instrument WingsLog (SquawkIt) across Android, iOS, and web with Firebase Analytics → GA4: automatic page-view tracking, manual-but-enforced button-click events, latency for attachment download & export, and app-version / platform distribution. Built to scale as new pages are added.

1. Context & Goals

WingsLog currently ships no product analytics. We want to understand real usage across the three targets without bolting on per-screen boilerplate. Concretely:

  • Page displays — every screen view, automatically, including future screens.
  • Button clicks — taps on actionable controls, with a stable, readable id.
  • Operation latency — attachment download speed and export generation time (with throughput/size context).
  • Install distribution — app version and platform (Android / iOS / web) breakdowns.
  • Login state — anonymous, Google, Apple, or other providers (future expansion), as a dimension on every event.
Scalability requirement New pages must be tracked with zero per-screen code. Page views are handled by one central navigation observer; only button clicks and latency call sites require explicit (but lightweight, enforced) instrumentation.

2. Backend choice — Firebase Analytics → GA4

"Google Analytics" for an app means Firebase Analytics feeding a GA4 property. It is the natural fit here, with near-zero new infrastructure:

  • The project is already a Firebase project (wingslog-9ca4e).
  • The iOS Xcode project already links the FirebaseAnalytics frameworks (iosApp.xcodeproj/project.pbxproj).
  • Android already uses the Firebase BOM; web already calls Firebase.initialize(...) in webApp/.../main.kt.

Android / iOS / web register as three data streams in one GA4 property, which makes platform distribution and app-version dimensions essentially free: GA4 auto-collects platform, app_version, and os_version on mobile. Web sets app_version explicitly. No raw GA4 Measurement Protocol is needed.

Why an abstraction is required GitLive's KMP Firebase SDK (used for Auth / Firestore / Storage / Functions) does not wrap Analytics. So core:analytics is a thin expect/actual layer over each platform's native Analytics SDK.

3. Architecture — core/analytics

A new core module core/analytics (plus a sibling core/analytics/lint), registered in settings.gradle.kts, following the canonical core-module layout.

3.1 commonMain API

interface AnalyticsManager {
  fun logScreenView(screenName: String, params: Map<String, String> = emptyMap())
  fun logClick(id: String, params: Map<String, String> = emptyMap())
  fun logTiming(operation: String, durationMs: Long, params: Map<String, String> = emptyMap())
  fun setUserProperty(name: String, value: String)
}

// Provided once at the Compose root, mirroring core/ui/adaptive's LocalLayoutTier.
val LocalAnalytics: ProvidableCompositionLocal<AnalyticsManager>

// Ergonomic, "normal onClick" helper — id is a required parameter.
@Composable
fun loggedClick(id: String, params: Map<String, String> = emptyMap(), onClick: () -> Unit): () -> Unit {
  val analytics = LocalAnalytics.current
  return { analytics.logClick(id, params); onClick() }
}

suspend inline fun <T> AnalyticsManager.measureTiming(
  operation: String, params: Map<String, String> = emptyMap(), block: () -> T,
): T {
  val start = TimeSource.Monotonic.markNow()
  val result = block()
  logTiming(operation, start.elapsedNow().inWholeMilliseconds, params)
  return result
}

An analyticsModule Koin module exposes single<AnalyticsManager>.

3.3 App version & platform

  • Platform distribution + mobile app version are auto-collected (three data streams).
  • Web sets app_version explicitly at init from core/appinfo jsMain GeneratedVersionInfo.kt via setUserProperty.
  • core/appinfo.getAppVersion() is @Composable; add a non-composable accessor (e.g. appVersionString()) so the version is readable at startup outside composition.

3.4 Login-state user property

Login state is set as the login_state user property, so it dimensions every event (page views, clicks, timing) without being attached per call. It is derived from the GitLive FirebaseUser that AppEntry.kt already observes via firebaseAuth.authStateChanged:

enum class LoginState { ANONYMOUS, GOOGLE, APPLE, OTHER }

fun FirebaseUser?.toLoginState(): String = when {
  this == null         -> "logged_out"
  isAnonymous          -> "anonymous"
  else -> when (val id = providerData.firstOrNull {
      it.providerId != "firebase" }?.providerId) {
    "google.com" -> "google"
    "apple.com"  -> "apple"
    null          -> "other"
    else          -> id   // future providers logged by raw id, no code change
  }
}

The mapping helper lives in core:analytics (or core:auth) — no UI dependency. AppEntry.kt already runs firebaseAuth.authStateChanged.collect { user -> ... } for navigation; the same observer calls analytics.setUserProperty("login_state", user.toLoginState()), so the property updates immediately on sign-in, sign-out, anonymous start, and account upgrade. It is also set once at startup from the current user.

Future expansion Unknown providers fall through to their raw Firebase providerId (e.g. microsoft.com, oidc.<name>), so adding a new sign-in method needs no analytics change. Promote a raw id to a friendly label in the when only when you want a cleaner GA dimension value.

4. Page-view tracking (hybrid, scalable)

A single root-NavController observer is not sufficient, because the app has two kinds of "screen":

SurfaceHow it switchesSeen by root observer?
Login, the shell container, secondary routes (SyncSettings, Export, FeatureLab, ManageTechnicians…)Real NavHost routes (composable(...))✅ Yes
Add/Edit Aircraft · Task · Log · Squawk — rendered as full screens (compact) or dialogs (large)dialog(...) routes — still real back-stack entries, same navigate() call on both form factors✅ Yes — dialogs emit on currentBackStackEntryFlow
Shell sections (Dashboard / Tasks / Squawks / Logs / Settings) and the active aircraftAmbient state — AdaptiveShellViewModel.selectSection() / selectAircraft() mutate _uiState; route stays adaptive_shell❌ No — all collapse into one adaptive_shell view
Sidebar-tier Settings sub-pagesA separate settingsNav NavController nested in SettingsSection (AppEntry.kt)❌ No — different NavController
The actual gap is state-driven screens, not dialogs. Dialogs are captured fine (they're back-stack destinations). The blind spot is the adaptive shell, which renders the app's most-used surfaces — the per-aircraft Dashboard/Tasks/Squawks/Logs and aircraft switching — as ViewModel state under one route.

4.1 Three feeders into one logScreenView

We keep one logging sink (analytics.logScreenView(name)) fed from three centralized places — no per-screen sprinkling:

  1. Root NavController observer — in AppEntry.kt, covers Login, dialogs, and compact-tier secondary routes. The adaptive_shell container route is suppressed here (feeder 3 is the source of truth for in-shell screens), to avoid a meaningless duplicate. 
    LaunchedEffect(navController) {
  navController.currentBackStackEntryFlow.collect { entry ->
    entry.destination.route
      ?.takeIf { it != Screen.AdaptiveShell.route }
      ?.let { analytics.logScreenView(it) }
  }
}
  2. Shell section + aircraft observer — the single funnel point is AdaptiveShellViewModel. Inject AnalyticsManager and emit a screen view whenever the section (or selected-aircraft context) changes, plus once for the initial state. Because every section/aircraft switch already routes through selectSection() / selectAircraft(), a new ShellSection is tracked automatically (screen name e.g. aircraft/tasks, aircraft/logs).
  3. Nested Settings NavController observer — attach the same observer helper to settingsNav in SettingsSection, so sidebar-tier Settings sub-pages register identically to the compact tier.

Factor the observer into a reusable helper (e.g. fun NavController.trackScreenViews(analytics, suppress: Set<String> = emptySet())) so feeders 1 and 3 share one implementation. Route templates come from core/nav/Screen.kt; the raw route is the screen name (optionally prettified via a small map later).

Scalability New full routes → tracked by feeder 1 with zero code. New shell sections → tracked by feeder 2 because all section changes funnel through selectSection(). The only deliberate decision per new surface is whether it's a route or shell state — both paths are already wired.

5. Button-click tracking & conformance

Call sites stay "normal onClick", with a manual log via loggedClick:

Button(onClick = loggedClick("ExportLogs/start_export") { viewModel.startExport() }) { ... }

Because loggedClick returns a plain () -> Unit, the call site reads like an ordinary onClick, but the id is a required parameter — you cannot build the handler without one.

5.1 Conformance — custom Lint rule

Nothing in the type system forces a bare onClick = { } to log. To enforce it, add a custom Android Lint rule in core/analytics/lint that plugs into the existing CI step (./gradlew lint — currently stock Android Lint, no baseline, no detekt/ktlint/spotless).

  • lint-type Gradle module (com.android.tools.lint:lint-api / lint-checks): an IssueRegistry + a Detector.UastScanner.
  • Flags clickable composables — Button, IconButton, TextButton, FilledTonalButton, OutlinedButton, FloatingActionButton, Modifier.clickable, combinedClickable — whose onClick/clickable lambda body contains no call to logClick / loggedClick.
  • Issue id MissingClickAnalytics, Severity.ERROR. Escape hatch: @Suppress("MissingClickAnalytics") for legitimate forwarding (onClick = vm::save, or passing onClick down to a child).
  • Wire with lintChecks(project(":core:analytics:lint")) on UI modules. Android Lint analyzes the Android variant, which compiles commonMain, so shared Compose code is covered.
Known limitation (documented in the rule) The detector is a lexical heuristic — it cannot follow a method reference to see whether logging happens deeper in the call chain. It accepts some false negatives; that is the cost of bare onClick without a mandatory wrapper component. The loggedClick helper keeps compliance to a one-liner.

7. Wiring (DI + platform init)

  1. Register analyticsModule in composeApp/.../di/initKoin.kt (Android + iOS) and webApp/.../main.kt.
  2. Initialize the platform Analytics handle where Firebase already starts: Android WingsLogApplication.onCreate (after FirebaseApp.initializeApp); iOS MainEntry.doInitKoin/doInitKoinDogfood; web main.kt after Firebase.initialize(...) (also set app_version there).
  3. Provide LocalAnalytics + add the nav observer in AppEntry.kt.

9. Conventions to honor

  • 2-space Kotlin indentation; kotlin.time (never kotlinx.datetime); typed get<T>() in Koin modules.
  • core:storage / core:ui already api-export shared deps — don't redeclare downstream.
  • Dependency rules: datamanager may depend on core:analytics (no UI). The loggedClick / LocalAnalytics Compose helpers live in the part of core:analytics that UI modules depend on (mirroring how core:ui exposes Compose utilities).

10. Verification

  1. ./gradlew assembleDebug — Android builds with the new module + analytics dep.
  2. ./gradlew lint — the MissingClickAnalytics rule registers and fires on a deliberately un-logged Button, then passes once loggedClick is added.
  3. ./gradlew testDebugUnitTest + :core:analytics:lint detector tests via Lint's lintProject harness.
  4. ./gradlew :webApp:jsBrowserDevelopmentWebpack — web builds; check GA DebugView for screen_view, click, and timing events.
  5. Android dogfood: adb shell setprop debug.firebase.analytics.app dev.fanfly.wingslog, watch Firebase DebugView — confirm switching shell sections (Dashboard → Tasks → Squawks → Logs) and aircraft each fire a distinct screen_view (not just one adaptive_shell), open an Edit dialog on a large-screen layout and verify it also fires, tap a tracked button (click), run an export + attachment download (timing), confirm platform + app_version + login_state populate (sign in with Google / Apple and re-check the property).
  6. iOS: run the iosAppDogfood scheme with -FIRDebugEnabled and confirm the same in DebugView.
