Deletion Cascade & Storage GC — Design

Epic: [P7] #155 · This doc: #156 · Sub-issues: #157 (cascade), #158 (blobs), #159 (sweep), #160 (local GC)

Implementation status — DESIGN ONLY. Nothing here is built except the aircraft→children cascade, which shipped as part of aircraft sharing (§2.4) and is reused rather than rewritten.

1. The problem

Deleting something in SquawkIt does not delete it. It writes a tombstone (deleted = true) and keeps the payload. That is correct for sync — a tombstone is how the deletion reaches other devices, and last-write-wins needs it to outlive any offline client — but nothing ever finishes the job:

  • Firestore tombstones accumulate forever. There is no server-side sweep. Every record the user has ever deleted still sits in their tree, payload intact.
  • Storage blobs orphan. Deleting a log, squawk, or task does not delete its attachments. Deleting an aircraft does not delete its blobs either.

Two costs. Money — bytes we pay for indefinitely, per user, growing monotonically. And trust — a user who deletes a photo of a damaged part has not deleted it; the bytes are still in Storage, retrievable by anyone who later gains read access to that path. "Delete" that does not delete is a promise we are not keeping.

2.4 What already exists

onAircraftDeleted already tombstones every child record on the deleted: false → true transition, discovering subcollections with listCollections() rather than a hardcoded kind list. It was written for the sharing teardown and it is exactly the cascade #157 asks for. #157 is therefore mostly done — what is missing is the blob half.

3. The crux: the server cannot read a record

Blobs are aircraft-scoped. The path is built from the owning record's EntityScope, and every attachment-bearing record (log, task, squawk) sits at aircraftChild(uid, acId), so every blob lands under its aircraft:

users/{uid}/aircraft/{acId}/blobs/{blobId}

Note what the path does not contain: the id of the record the attachment belongs to. It says which aircraft the blob is for, and nothing about which log or squawk owns it.

That mapping — record → its blobs — exists in exactly one place: inside the record's opaque protobuf payload. And a Cloud Function looking at a tombstoned log sees payload: <bytes>. It cannot tell which blobs just died with it.

This is the same wall the rules hit (they cannot read payloads either, which is why the sharing ACL exists as plain fields), and the same wall the invite preview hit (#201 — the aircraft label had to be denormalized onto the invite, because the server cannot read the registration out of the aircraft record).

4. Options considered

A. Decode payloads server-side

Generate TS types for the record protos (the functions already do this for two protos via generate:proto) and decode the payload in the trigger to find blob ids.

Rejected as the primary mechanism. It works, but it couples every Cloud Function to the full record schema — a proto change becomes a backend deploy, and a decode failure on an unexpected version means silently skipping cleanup (or, worse, deleting the wrong thing). It buys nothing that option C does not, at a much higher standing cost.

B. Client deletes its own blobs on record delete

The client can decode the payload — it wrote it. So deleteLog() could walk the attachments and delete each blob.

Necessary but not sufficient. The intent dies with the process: delete a log, kill the app before the blob delete drains, and the bytes are orphaned with nothing left that knows they should go. Uninstall the app and the intent is gone for good. A cleanup that only runs when the client survives is not a cleanup.

C. The client states the intent; the server carries it out ✅ RECOMMENDED

The client knows which blobs a record owns. The server can act durably. So the client writes the intent into the tombstone, as a plain field on the envelope:

SyncDocWire {
  payload: <bytes>        // opaque, as today
  deleted: true
  writerUid: "…"
  blobIds: ["a1b2…", "c3d4…"]   // NEW: plain field, set when tombstoning a record
}

A Firestore trigger on the deleted → true transition reads blobIds and deletes those Storage objects. No proto decoding server-side. The intent is durable the moment the tombstone is pushed — the client may die immediately after and the bytes still get collected. It is idempotent (deleting an absent object is a no-op) and it survives an uninstall.

D. Re-path blobs under the record id

users/{uid}/aircraft/{acId}/blobs/{recordId}/{blobId} would make the cascade a prefix delete with no payload knowledge at all.

Not now. It is the cleanest shape and worth adopting for new blobs, but it does not help the bytes already written, so it does not remove the need for C. Filed as a follow-up rather than smuggled into this epic.

5. Design

5.1 Record delete → blobs (#158)

  1. Client tombstones the record and stamps blobIds on the envelope (option C).
  2. Trigger onRecordDeleted fires on deleted: false → true for the per-aircraft kinds, reads blobIds, deletes each Storage object.
  3. The client also deletes its local copy (option B) so the device reclaims space immediately without waiting for a round trip.

5.2 Aircraft delete → blobs

No decoding and no blobIds needed: blobs are aircraft-scoped, so the whole users/{uid}/aircraft/{acId}/blobs/ prefix dies with the aircraft. Extend the existing onAircraftDeleted trigger with a prefix delete.

5.3 The scheduled sweep (#159)

A daily Cloud Function, over every user's tree. Two jobs:

  • Purge expired tombstones. Hard-delete Firestore docs with deleted = true older than the retention window. This is what stops unbounded growth.
  • Collect orphaned blobs. A backstop for bytes whose intent was lost — uploads that completed but were never referenced, and every blob orphaned before this epic ships (there will be a backlog).
Retention is a correctness constraint, not a cost knob. Purge a tombstone before every device has seen it and the record comes back: an offline client still holding the live row syncs, finds no tombstone, and pushes it up again as a fresh write. The window must comfortably exceed the longest plausible offline stretch. 30 days, matching the local TombstoneGc, so the two ends agree.

5.4 Safety

  • Never delete a blob a live record still references. The orphan sweep must prove a blob is unreferenced before deleting it — and since the server cannot read payloads, "unreferenced" can only be established from the blobIds index. Until that index has been populated for long enough to be trusted, the sweep runs in dry-run and reports what it would delete.
  • Idempotent and resumable. Deleting an absent object is a no-op; the sweep is paged and can be re-run.
  • Keyed by the tree it is sweeping. The #204 lesson: an aircraft id is unique only within a user's tree. Every path this touches is built from the uid it is sweeping, never from an id supplied elsewhere.

5.5 Sharing interplay

A revoked member's client purges its local copy (SharedScopeJanitor) and must never delete host-owned bytes — it does not, and must not start. Deletion of the canonical data happens only in the host's tree, driven by the host's own delete. This matters more once blobs are shared ([P8] #196): the sweep must be written so that a member's action can never reach a host's Storage.

7. Open questions

  1. Is 30 days right? It is the longest an offline device can be away before its deletions (or its resurrections) go wrong. A pilot who flies for a season without opening the app is not an unusual user.
  2. The existing backlog. Every blob orphaned to date has no blobIds record pointing at it. Collecting those needs either a one-off payload-decoding pass (option A, run once, offline) or accepting the loss and starting the accounting from now. Decide before #159 is armed.
  3. Does deletion need to be provable? If a user asks "is my photo really gone", the honest answer today is "eventually, if a sweep runs". A hard-delete-on-request path (GDPR-shaped) is a different feature, and worth naming as one.
