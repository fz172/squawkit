Deletion Cascade & Storage GC — Design

Epic: [P7] #155 · This doc: #156 · Sub-issues: #157 (cascade), #158 (blobs), #159 (sweep), #160 (local GC)

Implementation status — SHIPPED. The cascade (#157), blob cleanup on delete (#158), the scheduled sweep (#159, armed — STORAGE_SWEEP_DRY_RUN=false) and the local TombstoneGc (#160) are all in main. Retention agreement between the two ends is enforced by TombstoneRetentionAgreementTest, which reads the sweep's own .env. One deliberate omission: the client still reclaims a deleted record's local blob at GC time, not the moment the record is deleted — the §5.1 "option B" optimisation, which buys back disk sooner but changes nothing about what is or is not deleted.

1. The problem

Deleting something in SquawkIt does not delete it. It writes a tombstone (deleted = true) and keeps the payload. That is correct for sync — a tombstone is how the deletion reaches other devices, and last-write-wins needs it to outlive any offline client — but nothing ever finishes the job:

  • Firestore tombstones accumulate forever. There is no server-side sweep. Every record the user has ever deleted still sits in their tree, payload intact.
  • Storage blobs orphan. Deleting a log, squawk, or task does not delete its attachments. Deleting an aircraft does not delete its blobs either.

Two costs. Money — bytes we pay for indefinitely, per user, growing monotonically. And trust — a user who deletes a photo of a damaged part has not deleted it; the bytes are still in Storage, retrievable by anyone who later gains read access to that path. "Delete" that does not delete is a promise we are not keeping.

2. What actually happens today

2.4 What already exists

onAircraftDeleted already tombstones every child record on the deleted: false → true transition, discovering subcollections with listCollections() rather than a hardcoded kind list. It was written for the sharing teardown and it is exactly the cascade #157 asks for. #157 is therefore mostly done — what is missing is the blob half.

3. What the server can and cannot read

Blobs are aircraft-scoped. The path is built from the owning record's EntityScope, and every attachment-bearing record (log, task, squawk) sits at aircraftChild(uid, acId), so every blob lands under its aircraft:

users/{uid}/aircraft/{acId}/blobs/{blobId}

Note what the path does not contain: the id of the record the attachment belongs to. It says which aircraft the blob is for, and nothing about which log or squawk owns it. That mapping lives inside the record's protobuf payload.

It is tempting to call that a wall. It is not — and the distinction matters:

Can decode a payload?
Security rulesNo. Rules see opaque bytes. This is why the sharing ACL exists as plain rules-readable fields at all.
Cloud FunctionsYes. They already do — generate:proto emits TS types from the same .proto files the client compiles, and two protos are wired through it today.

And the tombstone the client pushes carries everything a decoder needs:

SyncDocWire {
  payload: <bytes>              // retained on delete — the record is still in there
  schema:  "aircraft.MaintenanceLog"   // names the type, so a trigger can dispatch
  deleted: true
}

So when the client says "delete log XYZ", the server can read XYZ, decode it, and pull out the attachment ids. Nothing blocks it.

4. Options considered

A. Decode the payload server-side ✅ RECOMMENDED

A trigger on the deleted: false → true transition reads the tombstoned doc, dispatches on schema, decodes the payload, and deletes the Storage objects for the attachments it names.

  • No wire change. Nothing to add to the envelope, nothing to migrate.
  • Works for records deleted by old clients. The cleanup does not depend on the deleting client being new enough to have cooperated.
  • Covers the existing backlog. Every blob orphaned to date is still described by a tombstone that is still in Firestore. The sweep can decode those and collect the bytes — which no client-side scheme can do, because those clients are long gone.
  • One source of truth. The payload is the reference list.

B. Client deletes its own blobs on record delete

The client can decode its own payload, so deleteLog() could delete each blob.

Keep, but only as a local optimisation. It reclaims device space immediately without a round trip. It cannot be the mechanism: the intent dies with the process. Delete a log, kill the app before the blob delete drains, and the bytes are orphaned with nothing left that knows they should go. Uninstall, and the intent is gone for good.

C. Client stamps blobIds onto the tombstone envelope

Add a plain field listing the record's blobs, so the server needs no decoder.

Rejected. An earlier draft of this doc recommended it, on the false premise that Cloud Functions could not decode payloads. They can. Once that is untrue, this is strictly worse: it needs a wire change, it does nothing for records deleted by clients that predate it, it cannot touch the existing backlog, and it introduces a second source of truth — a denormalized list that can drift from the payload it describes. A drifted list either leaks bytes or, worse, deletes a blob a live record still uses.

D. Re-path blobs under the record id

users/{uid}/aircraft/{acId}/blobs/{recordId}/{blobId} would make the cascade a prefix delete needing no payload knowledge at all.

Not now. It is the cleaner shape and worth adopting for new blobs, but it does nothing for the bytes already written, so it does not remove the need for A. Filed as a follow-up rather than smuggled into this epic.

5. Design

5.1 Record delete → blobs (#158)

  1. Client tombstones the record, as today. No client change is required for correctness.
  2. Trigger onRecordDeleted fires on deleted: false → true for users/{uid}/aircraft/{acId}/{kind}/{docId}, dispatches on the envelope's schema, decodes the payload, and deletes the Storage object for each attachment it names.
  3. Before deleting a blob, confirm no other live record in the same aircraft references it. Attachment ids are per-attachment, but a copy/duplicate feature could put the same id on two records, and deleting a blob a live record still shows is unrecoverable.
  4. A decode failure (unknown schema, corrupt bytes) skips and alerts. It never guesses, and it never deletes.
  5. The client drops its local blob copy when the tombstone is purged (#160). Doing it the moment the record is deleted, so the device reclaims space immediately (option B), remains an optimisation — not the mechanism, and not built.

5.2 Aircraft delete → blobs

No decoding and no blobIds needed: blobs are aircraft-scoped, so the whole users/{uid}/aircraft/{acId}/blobs/ prefix dies with the aircraft. Extend the existing onAircraftDeleted trigger with a prefix delete.

5.3 The scheduled sweep (#159)

A daily Cloud Function, over every user's tree. Two jobs:

  • Purge expired tombstones. Hard-delete Firestore docs with deleted = true older than the retention window. This is what stops unbounded growth.
  • Collect orphaned blobs. A backstop for bytes whose intent was lost — uploads that completed but were never referenced, and every blob orphaned before this epic ships (there will be a backlog).
Retention is a correctness constraint, not a cost knob. Purge a tombstone before every device has seen it and the record comes back: an offline client still holding the live row syncs, finds no tombstone, and pushes it up again as a fresh write. The window must comfortably exceed the longest plausible offline stretch. 30 days, matching the local TombstoneGc, so the two ends agree.

5.4 Safety

  • Never delete a blob a live record still references. "Unreferenced" is established by decoding the live records in that aircraft — the payloads are the reference list, so the question is answerable exactly rather than inferred. The sweep still runs in dry-run first and reports what it would delete: the failure mode here is destroying a user's photos, and no amount of confidence in the logic justifies skipping the rehearsal.
  • Idempotent and resumable. Deleting an absent object is a no-op; the sweep is paged and can be re-run.
  • Keyed by the tree it is sweeping. The #204 lesson: an aircraft id is unique only within a user's tree. Every path this touches is built from the uid it is sweeping, never from an id supplied elsewhere.

5.5 Sharing interplay

A revoked member's client purges its local copy (SharedScopeJanitor) and must never delete host-owned bytes — it does not, and must not start. Deletion of the canonical data happens only in the host's tree, driven by the host's own delete. This matters more once blobs are shared ([P8] #196): the sweep must be written so that a member's action can never reach a host's Storage.

#160Done. The local TombstoneGc now reclaims the blobs a purged tombstone referenced — decoding the payload before it deletes the row, since the payload is the only record of which blobs were the record's. A purged aircraft takes every blob under its scope, which also collects the ones no surviving payload names. It never touches the remote object: those bytes are the server's to delete, and on a shared aircraft they are the host's (§5.5). A blob a live record still names is kept. Retention agreement with the server is a test, not a comment.

7. Open questions

  1. Is 30 days right? It is the longest an offline device can be away before its deletions (or its resurrections) go wrong. A pilot who flies for a season without opening the app is not an unusual user.
  2. The existing backlog. Answered by option A. Every blob orphaned to date is still described by a tombstone still sitting in Firestore, so the sweep can decode those and collect the bytes. This was an open question only while the design assumed the server could not read a payload.
  3. Does deletion need to be provable? If a user asks "is my photo really gone", the honest answer today is "eventually, if a sweep runs". A hard-delete-on-request path (GDPR-shaped) is a different feature, and worth naming as one.
