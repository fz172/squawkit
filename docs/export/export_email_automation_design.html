Design Doc: Automatic Email Delivery for Export

Extends docs/export/export_logs_design.md with automatic delivery after export generation, a reminder path when no email is known, and an export history model that can represent both local and remote archives across devices.

1. Problem

The current export flow ends with a local file and a manual share action. That is adequate when the user is on the same device and wants to pick an email app themselves, but it breaks two product goals:

  1. The app cannot complete the handoff automatically even when it already knows where the export should be sent.
  2. The export history only reflects files discoverable on the current device, so a user on a second phone cannot see that an export already happened elsewhere.

We need a delivery model where export generation remains device-local, but delivery and history metadata are durable at the account level.

2. Product Decision

Case Behavior
User email is known Generate the export as usual, upload/archive it, then automatically send the delivery email when export processing completes.
User email is not known Keep the current success screen and manual share flow, but add a reminder on the export ready screen telling the user to update their email address in Backup & Sync settings.
Export history Keep showing all past exports, but each row may represent either a local archive on this device or a remote archive created elsewhere.

3. Goals and Non-Goals

Goals

  • Automatically send the export email when a deliverable email address is already known.
  • Preserve the current manual share escape hatch when no email is known.
  • Make export history account-scoped instead of device-scoped.
  • Allow history rows to clearly distinguish local versus remote archive availability.
  • Allow signed-in users to delete the local file, remote blob, and export record as one action.
  • Do this without changing the logbook export archive format itself.

Non-Goals

  • Replacing the existing ZIP/PDF/XLSX/CSV export generation pipeline.
  • Building a general outbound email platform for arbitrary app messages.
  • Supporting automatic delivery or remote export history for signed-out users.
  • Supporting anonymous users with automatic email send.

4. Proposed User Experience

4.1 Export selection and run

The export selection flow in feature/export/update/ExportSelectionScreen.kt stays unchanged until completion. The main branching begins only after archive creation.

4.2 Success state when email is known

  • The app still shows the export-ready result screen.
  • The primary message changes from "share this file" to "email sent" or "sending email" depending on delivery state.
  • The user can still open history and manually share the local file if it exists on the current device.
  • If automatic send fails, the result screen degrades to a retry/manual-share recovery state instead of pretending success.

4.3 Success state when email is unknown

  • Keep the current share-first action set.
  • Add a reminder card on the success screen: "Add your email in Backup & Sync to send future exports automatically."
  • The reminder deep-links to Screen.SyncSettings.

4.4 Export history

  • History remains the single list of past exports.
  • A local item can be shared or deleted on this device.
  • A remote item was generated earlier for the same account but its file is not present on this device.
  • Remote items still show timestamp, aircraft/date-range summary, and delivery status.
  • Remote-only rows support delete in the first release; delete removes the remote blob and the export record.

6. Source of Truth for Email Address

The current SyncSettings feature has no editable email field. This design adds one explicitly instead of inferring that FirebaseAuth.currentUser.email is always sufficient.

Resolution rules

  1. Use an explicit export-delivery email saved in Backup & Sync settings if present.
  2. Otherwise, fall back to the signed-in auth email when non-blank and verified enough for product policy.
  3. If neither exists, treat the user email as unknown and keep the current manual flow.

The settings UI should expose a single editable row under Backup & Sync, phrased as the destination for automatic exports. The doc does not require a broader profile-edit redesign; this can live beside the existing sync toggles.

This is a strictly signed-in feature. Signed-out users cannot edit the destination email, cannot upload remote export blobs, and cannot use automatic delivery.

7. Data Model

7.1 Evolve the existing device-local manifest

The current proto in core/model/src/commonMain/proto/export/export_record.proto is already richer than a thin file listing. It persists:

  • file_path, file_name, size_bytes, and creation time
  • display_location
  • formats
  • a flattened display-ready date_range
  • a display-ready aircraft snapshot list

The current implementation also uses an app-private ExportRecordIndex as a metadata manifest and reconciles it against archives discovered on disk. In other words, today's design is already:

  • manifest-backed for scope metadata
  • local-device-only for existence
  • keyed by file_path, which is not stable across devices

The design change needed for automatic delivery is therefore not "add metadata from scratch." It is "evolve the existing local manifest into an account-scoped manifest that can survive across devices and delivery states."

7.2 Proposed additive shape

The most important missing fields are a stable export identifier and backend delivery metadata. The existing scope fields should be preserved rather than replaced. 

data class ExportRecord(
  val exportId: String,
  val fileName: String,
  val sizeBytes: Long,
  val createdAtEpochMillis: Long,
  val displayLocationKind: ExportDisplayLocation,
  val availability: ExportAvailability,
  val delivery: ExportDeliveryState,
  val destinationEmail: String?,
  val aircraftSummary: List<String>,
  val dateRangeSummary: String,
  val localFilePath: String?,
  val remoteArchiveRef: String?,
)

enum class ExportAvailability {
  LOCAL,
  REMOTE,
  LOCAL_AND_REMOTE,
}

sealed interface ExportDeliveryState {
  data object NotRequested : ExportDeliveryState
  data object Queued : ExportDeliveryState
  data object Sending : ExportDeliveryState
  data class Sent(val sentAtEpochMillis: Long) : ExportDeliveryState
  data class Failed(val reason: String) : ExportDeliveryState
}

7.3 Proto implications

Based on the current proto and implementation, the backend-ready version should be treated as an additive schema evolution, not a separate unrelated record type.

  • Keep the current display-ready fields: formats, date range, and aircraft summary.
  • Add a stable export_id because file_path is only a local join key.
  • Add backend-oriented fields such as remote archive reference, destination email snapshot, and delivery state.
  • Allow records to remain visible even when the local file is absent, which is the opposite of the current prune-on-missing behavior.

7.4 Why both local and remote references matter

  • localFilePath is needed for share/open/delete actions on the current device.
  • remoteArchiveRef is needed so another device can still show the export in history.
  • availability prevents UI code from guessing from null fields repeatedly.

8. Storage and Sync Model

Local

Keep writing the archive to the platform-visible location using the existing ExportFileStore actuals. That preserves manual share and offline access on the generating device.

Remote

Add a lightweight export manifest collection plus remote blob storage. The manifest is account-scoped metadata; the blob is the optional downloadable archive payload.

Layer Proposed shape
Manifest metadata users/{uid}/export_history/{exportId} or equivalent sync-backed collection with timestamp, filename, summaries, delivery state, and remote blob reference.
Archive blob Remote object storage entry keyed by uid/exportId/fileName.
Local mirror SQLDelight entity table for export manifests so history loads offline after the first sync.

This is intentionally the same pattern already used elsewhere in the local-first system: user-visible metadata should sync as structured entities, while large binary payloads live in blob storage.

One behavior must change from the current implementation: today missing local archives are pruned from history when the on-device index is reconciled. For cross-device history, the synced manifest becomes the source of truth for record existence, while the local file system only answers whether this device currently has a downloadable/shareable copy.

9. Firebase Backend Design

Automatic delivery is not a client-only feature. The app needs Firebase Storage for the archive blob, Firestore for manifest state, and a Firebase Cloud Function to perform the trusted send operation.

9.1 Proposed Firebase pieces

Firebase service Role
Firebase Auth Defines the owning uid. Automatic delivery and remote export history are allowed only for signed-in non-anonymous users.
Firebase Storage Stores the generated export archive at a stable per-user object path.
Cloud Firestore Stores the export history manifest and delivery state so all devices can observe it.
Firebase Cloud Functions Runs the trusted backend send flow and updates delivery state transitions.

9.2 Blob and manifest shape

Storage:
  gs://<bucket>/users/{uid}/exports/{exportId}/{fileName}

Firestore:
  users/{uid}/export_history/{exportId}

The archive object path mirrors the existing repo convention of user-scoped storage paths. The Firestore document holds lightweight metadata only, including filename, size, creation time, aircraft/date-range summary, destination email snapshot, and delivery state.

9.3 Recommended Cloud Function contract

Use a callable function for the initial implementation: requestExportDelivery(exportId).

  1. The client generates the archive locally.
  2. The client uploads the archive to Firebase Storage using the signed-in user's credentials.
  3. The client writes or updates the Firestore manifest with delivery = Queued.
  4. The client calls the Cloud Function with only the exportId.
  5. The Cloud Function derives uid from the Firebase auth context, loads the manifest, validates ownership, validates destination email, and performs the send.

The client should not pass raw storage paths or arbitrary destination emails as trusted send parameters when the server can derive them from owned manifest state.

9.4 Why callable instead of Storage-triggered first

  • It gives the client an explicit transition point after upload and manifest write.
  • It keeps the server logic keyed by a stable business ID, exportId, instead of reverse-parsing object paths.
  • It is easier to make idempotent and easier to retry intentionally from the UI.
  • It avoids triggering sends for every object upload under a broad storage prefix.

9.5 Cloud Function responsibilities

  1. Authenticate the caller and reject anonymous or missing auth context.
  2. Load users/{uid}/export_history/{exportId}.
  3. Verify the manifest is in a sendable state and has a valid remoteArchiveRef.
  4. Transition Firestore state from Queued to Sending.
  5. Fetch the archive bytes or generate a short-lived signed download URL for the mail provider.
  6. Send the email through the chosen provider.
  7. Write back Sent or Failed with timestamp and server-side error details safe for display.

9.6 Mail delivery mode

The backend email always uses a signed download link. The server never attaches archive files directly to the outbound email.

This avoids mail-provider attachment limits, reduces Cloud Function memory/runtime pressure, and keeps delivery behavior consistent regardless of export size.

9.6a Remote retention policy

Remote export archive blobs are retained for 60 days. The manifest should record the remote expiry time so the UI can explain when a remote archive is no longer expected to be downloadable.

9.7 Idempotency

The Cloud Function must be idempotent. Repeated calls for the same exportId must not send duplicate emails once a manifest is already marked Sent. 

if delivery == Sent:
  return success
if delivery == Sending and lease not expired:
  return in_progress
if delivery == Queued or retryable Failed:
  acquire lease and continue

A simple lease field such as deliveryLeaseExpiresAt is enough to prevent two concurrent send attempts from racing.

9.8 Security rules boundary

  • Firebase Storage rules should allow a user to write only under users/{uid}/exports/*.
  • Firestore rules should allow a user to read their own export history documents.
  • Clients should not be allowed to mark a delivery as Sent; only the backend should write terminal delivery success state.
  • Destination email should be treated as user-owned input, but final send eligibility should still be validated server-side.

9.9 Recommended Firestore document shape

{
  exportId: string,
  uid: string,
  fileName: string,
  sizeBytes: number,
  createdAt: timestamp,
  aircraftSummary: string[],
  dateRangeSummary: string,
  localGenerated: boolean,
  remoteArchiveRef: string | null,
  destinationEmail: string | null,
  destinationEmailSource: "SYNC_SETTINGS" | "AUTH_PROFILE" | null,
  deliveryState: "NOT_REQUESTED" | "QUEUED" | "SENDING" | "SENT" | "FAILED",
  deliverySentAt: timestamp | null,
  deliveryFailureCode: string | null,
  deliveryFailureMessage: string | null,
  deliveryLeaseExpiresAt: timestamp | null
}

The local app can map this into the richer KMM ExportRecord model by combining synced manifest state with local file existence checks.

10. Backend Code Location in This Repo

This repository is currently structured as a mobile application monorepo. The top-level directories are app/runtime concerns such as app/, composeApp/, core/, feature/, iosApp/, and docs/. There is no existing backend or Firebase Functions workspace.

10.1 Recommendation

Add a new top-level backend/ workspace, not a KMM feature module.

This is the cleanest fit for the current repo because Cloud Functions are deployment infrastructure, not runtime app code. They should live beside the app, not inside feature/export or core.

10.2 Why not put backend code under feature/export

  • feature/export follows the app's canonical KMM feature-module pattern and is built into the client.
  • Firebase Cloud Functions will likely be TypeScript or JavaScript, with its own package manager, lockfile, emulator config, and deployment lifecycle.
  • Mixing server code into the KMM module tree makes build ownership and dependency boundaries unclear.
  • The backend will serve account-level export history and delivery state, which are broader than one UI module.

10.3 Why not put backend code under core/

  • core/ in this repo is shared app code consumed by Android and iOS targets.
  • Cloud Functions do not participate in the KMM compilation model.
  • Keeping backend code outside core/ avoids implying shared runtime or binary linkage that does not exist.

10.6 Shared schema ownership

The client KMM code and the Cloud Function will both depend on the same conceptual export manifest schema, but this repo does not currently have a cross-language schema generation layer for Kotlin and TypeScript. For the first implementation:

  1. Define the canonical schema in this design doc and mirror it manually in Kotlin and TypeScript.
  2. Keep field names stable and flat.
  3. Avoid clever nested structures until the backend shape settles.

If the backend surface grows later, we can introduce a generated JSON-schema or protobuf contract, but that would be overkill for the first export-delivery function.

10.7 Deployment ownership

  • The mobile app continues to build with Gradle and Xcode from the existing repo root.
  • The Firebase backend gets its own Node/TypeScript toolchain under backend/firebase/functions.
  • Firebase rules and function deployment commands should run from the repo root or the backend/ workspace, not from any KMM module.
  • CI can later add a separate backend validation job without entangling it with Android/iOS compilation.

10.8 Alternative: separate backend repo

A separate repo is defensible if the backend team or deployment cadence is fully independent. Based on the current structure, though, that would introduce more overhead than value for this feature: two repos, duplicated issue context, and harder cross-review between client manifest changes and backend delivery logic.

For this project's current size, a top-level backend/ workspace inside this repo is the better default.

10.9 Public repo and secrets handling

Adding a backend/ workspace to a public GitHub repo does not by itself leak credentials. The risk comes from what is committed inside it.

Safe to commit Must not be committed
firebase.json, .firebaserc project aliases, firestore.rules, firestore.indexes.json, storage.rules, package.json, tsconfig.json, Cloud Function source code service account JSON keys, .env files with secrets, mail-provider API keys, SMTP passwords, private tokens, downloaded Admin SDK credentials

Firebase project IDs are usually not secrets. Client Firebase config values are also usually not secrets. The sensitive material is privileged backend credentials and third- party provider secrets.

10.10 Recommended secret storage model

  1. Commit backend source and Firebase config normally.
  2. Store mail-provider secrets in Firebase Functions secret/env management, not in the repo.
  3. Load secrets in function code from the runtime environment only.
  4. Keep local developer secret files ignored by git.
  5. Never commit a service-account key file for deployment or local testing.

    functions/
      .env.example        # placeholders only, safe to commit
      .gitignore          # ignores real .env files and local secret material

11. Delivery Flow

1. User taps Export
2. ExportManager builds archive locally
3. ExportHistoryRepository creates manifest: availability = LOCAL, delivery = NotRequested
4. If a destination email exists:
   a. ExportDeliveryManager uploads archive blob
   b. Manifest becomes availability = LOCAL_AND_REMOTE and writes remoteArchiveRef
   c. Client marks delivery = Queued
   d. Client calls Firebase callable function requestExportDelivery(exportId)
   e. Cloud Function moves delivery state Queued -> Sending -> Sent or Failed
5. If no destination email exists:
   a. No automatic send request is made
   b. Manifest remains local-only
   c. Success UI shows reminder to update email in Backup & Sync

Important sequencing rule

The email should be sent only after the archive blob is durably accessible to the backend mail sender. Otherwise we risk sending the user a broken link or an attachment request pointing at a local-only file.

12. UI Changes

12.1 Backup & Sync settings

  • Add an editable email destination row below the existing sync toggles.
  • Show resolved source when helpful: explicit export email versus signed-in account email.
  • Keep this field in feature/sync/settings rather than inventing a second settings surface.
  • Hide or disable this row when the user is signed out, since automatic export delivery is a signed-in-only feature.

12.2 Export success screen

Condition Result screen behavior
Email known, delivery queued/sent Show delivery status card. Primary action becomes View exports or Done; manual share is secondary.
Email unknown Keep current manual share primary action and add reminder card linking to Backup & Sync.
Email known, delivery failed Show failure banner with Retry send and manual share fallback.

12.3 Export history screen

  • Each row shows one of: Local, Remote, or Local + Remote.
  • Each row also shows delivery state: sent, pending, failed, or manual-only.
  • For remote-only rows, hide local share actions but still allow delete.
  • For local rows, keep the current share/delete actions.
  • Delete means best-effort removal of the local file, remote blob, and synced export record as one user action.

14. Failure Handling

Failure Expected behavior
Archive generation fails Current export error path. No success record.
Local file write fails Export fails; we should not send email for an archive we cannot fully finalize locally in this release.
Remote upload fails Success screen shows local export ready but automatic delivery failed. History row becomes LOCAL + Failed only if local write succeeded.
Email send fails after upload History row stays visible with Failed; user can retry send later or manually share the local archive.
User opens history on another device before blob download support exists Show remote metadata row with no local share action, but still allow remote delete.
Delete requested for a remote-only row Delete the synced export record and remote blob; local deletion is a no-op on this device because no local file exists.

15. Testing

  • Unit tests for email resolution precedence: explicit settings email wins over auth email.
  • Unit tests for manifest merge logic: local-only, remote-only, and local-plus-remote collapse into one row per export ID.
  • ViewModel tests for success UI branching when email is known versus unknown.
  • Delivery manager tests for manifest state transitions: NotRequested -> Queued -> Sent and failure variants.
  • History screen tests for action availability based on ExportAvailability.
  • Deletion tests for local-only, remote-only, and local-plus-remote records.
  • Cloud Function tests for auth rejection, ownership validation, idempotent resend suppression, and signed-link generation.

16. Rollout Plan

  1. Add manifest-backed history and remote upload plumbing first, behind a feature flag.
  2. Add the export-setup email field and export success reminder.
  3. Enable automatic send only for signed-in users with a resolved email address.
  4. Ship remote-delete support in the first release; remote download/recover can remain a follow-up if needed.

17. High-Level Implementation Plan by Milestone

17.1 Milestone 0: Backend foundation and callable hardening

Establish the Firebase Functions workspace as production infrastructure before export email logic lands.

  • Keep the existing backend/firebase/ workspace as the canonical server root.
  • Add the real export-delivery callable skeleton beside the current health probe.
  • Finalize App Check, auth, and allowed-app enforcement for callable access.
  • Add deployment documentation, secret placeholders, and runtime/version policy.

Exit criteria: backend repo layout, deploy path, auth/app-check policy, and secret handling are stable enough that the client can safely start integrating against a trusted callable boundary.

Status: Complete.

17.2 Milestone 1: Manifest-backed export history on device

Introduce the export-history domain first, without automatic email send, so later stages can attach delivery state to a durable business record instead of to transient UI state.

  • Add the synced/local export record model in feature/export.
  • Persist one stable exportId per successful local export.
  • Build list/read/delete plumbing for local-only records and merged local-plus-remote records.
  • Keep current manual share behavior unchanged for the user-facing success flow.

Exit criteria: the app can show export history from stable records, and delete/share no longer depends on one-shot success-screen state.

Status: Complete.

17.3 Milestone 2: Email destination source of truth and UI readiness

Add the product decision points around destination email before introducing backend send attempts.

  • Add an editable destination email field in export setup.
  • Implement resolver precedence: explicit settings email first, auth email second.
  • Update export success UI to branch between "email will be sent" and "manual share only".
  • Keep this strictly signed-in; anonymous users stay on the manual export path.

Exit criteria: the client can reliably decide whether automatic delivery should be requested and can explain that decision in UI.

Status: Complete.

17.4 Milestone 3: Remote upload and manifest sync

Teach the client to make an export backend-visible before any email send is attempted.

  • Upload completed export archives to Firebase Storage under the user-scoped path.
  • Write or update the Firestore manifest with remote archive reference and delivery snapshot fields.
  • Record initial delivery states such as NOT_REQUESTED and QUEUED.
  • Expose remote-aware history state in the export history UI.

Exit criteria: a completed export has both a local artifact and a backend-readable manifest row, even if automatic send is still disabled.

Status: Complete. Remote-backed records now upload to Firebase Storage, write a Firestore manifest snapshot, and render cloud-aware history state while remote delete remains deferred to Milestone 5.

17.5 Milestone 4: Server-driven automatic email delivery

Introduce the end-to-end automatic send only after local export, upload, and manifest state are durable.

  • Implement requestExportDelivery(exportId) in Cloud Functions.
  • Add backend validation for manifest ownership, sendable state, and destination email.
  • Generate signed download links and send through the chosen mail provider.
  • Write back terminal delivery states and retry-safe failure metadata.

Exit criteria: a signed-in user with a resolved destination email can complete a local export and receive a backend-generated delivery email without manual share.

Status: Complete. The app now queues delivery through the callable, the backend sends signed-link email via the configured provider, and manifest state moves through QUEUED, SENDING, SENT, or FAILED.

17.6 Milestone 5: Recovery, delete, and operational hardening

Finish the first production-ready slice by making the system manageable after the happy path succeeds.

  • Add retry behavior for failed delivery states from history.
  • Ship end-to-end delete for local files, remote blobs, and manifests.
  • Enforce remote retention and expiry messaging in the history UI.
  • Add monitoring, structured logs, and mail-provider failure inspection for operations.

Exit criteria: delivery failures are visible and recoverable, deletes are coherent across local and remote state, and the backend is operable in production.

Status: Complete. Export history now supports retrying failed delivery, delete removes local and remote export state coherently, the UI surfaces remote expiry, and the backend emits structured delivery logs for operations.

17.7 Suggested shipping sequence

  1. Ship Milestones 1 and 2 together behind a feature flag if the team wants user-visible history before backend delivery.
  2. Land Milestone 3 in dogfood/internal testing first to validate upload and manifest correctness.
  3. Enable Milestone 4 only after mail-provider secrets, signed-link generation, and idempotency are proven in staging-like usage.
  4. Treat Milestone 5 as release hardening, not an optional polish pass, because remote delete and failure recovery are part of the promised product behavior.

18. Resolved Product Decisions

  1. The backend email always uses a signed download link. The server never attaches archive files directly.
  2. Remote archive blobs are retained for 60 days.
  3. Automatic delivery and destination-email editing are strictly signed-in features.
  4. Delete removes the local file, remote blob, and export record; remote-only rows must support delete in the first release.

19. Recommended Scope for First Implementation

The narrowest coherent implementation is:

  1. Add a manifest-backed export history model with LOCAL and REMOTE availability.
  2. Add a destination email field under Backup & Sync plus auth-email fallback.
  3. After local export completes, upload the archive to Firebase Storage and trigger a callable Cloud Function only when a destination email resolves.
  4. Keep manual share intact as the fallback path and as the primary action when no email is known.
  5. Implement signed-link-only delivery, 60-day remote retention, and end-to-end delete in the first backend-backed release.

That yields the requested behavior without forcing a broader export format redesign.
