WingsLog · Hopply — Design / Implementation Plan

Adaptive Layout Redesign
Web & Large-Screen Tablet

Move WingsLog from a single-column, phone-shaped Compose UI to a genuinely adaptive shell — persistent sidebar on desktop, navigation rail on tablet, bottom bar on phone — driven by the Hopply Web design handoff. This document covers what to reuse, refactor, and remove in the current layout system, and how to switch between the mobile single-column layout and the adaptive system at runtime.

Status: M5 landed; M6 planned Target: web + tablet (iPad, Android tablets) Compose Multiplatform 1.11 · Material3 1.9 Source: Hopply Web.html handoff

This plan describes layout architecture, not pixel-level styling. The visual language (Space Grotesk + JetBrains Mono, blue primary, red/amber/green status, 12px cards, both themes) already exists in WingslogTheme and the design handoff — the work here is the adaptive shell, the navigation model, and the multi-pane surfaces.

0TL;DR

  • One new shell, built on NavigationSuiteScaffold. Wrap the authenticated app in Material3’s NavigationSuiteScaffold, which renders the navigation container as bottom bar → navigation rail → permanent drawer automatically by window size. We accept its default drawer width for the initial refactor (we are not reproducing the prototype’s bespoke 264px sidebar yet — see D2).
  • The navigation model flips — but only above phone. On tablet/desktop the selected aircraft becomes an ambient app-level context chosen from a switcher (in the top bar), with sections (Dashboard / Tasks / Logs / Squawks) scoped to it, plus global Settings. On phone there is no room for a switcher, so COMPACT keeps a dedicated fleet landing page exactly like the app today (aircraft list → tap → scoped sections). This tier-dependent root is the single biggest change.
  • Detail & edit flows become panes/drawers on large screens, staying as full-screen pushes / bottom sheets on compact. Same composable content, different container.
  • The data/domain layer is untouched. Local-first EntityStore, every ViewModel, and the sync engine carry over verbatim — this is a UI-shell change.
  • Add three dependencies: material3-window-size-class, material3-adaptive, material3-adaptive-navigation-suite (all multiplatform).
  • Reuse the most, remove the least. Screen content composables survive; the per-screen Scaffold/TopAppBar/FAB chrome and the flat NavHost graph are what gets refactored.

2Current layout system — audit

What actually exists in the codebase today (verified against AppEntry.kt, core/ui/.../navigation/Screen.kt, and a repo-wide search):

2.1 Navigation: a flat stack of full-screen destinations

AppEntry.kt hosts a single NavHost with four nested graphs:

// composeApp/.../AppEntry.kt
NavHost(navController, startDestination = GRAPH_AUTH) {
  authGraph(navController)       // Login / AuthFlow
  fleetGraph(navController)      // Dashboard (aircraft list) → Add/Edit aircraft
  aircraftGraph(navController)   // maintenance_overview/{aircraftId} → add/edit task,log,squawk
  settingsGraph(navController)   // Settings → Sync, Export, History, Technicians, FeatureLab
}

The flow is strictly hierarchical and back-stack driven: Dashboard (fleet) → tap aircraft → MaintenanceOverview/{aircraftId} (tabs inside the screen) → push maintenance_task_edit/{aircraftId}/{cardId}, etc. Aircraft identity is carried as a route argument, re-parsed on every destination.

2.2 The only “adaptivity” today is max-width capping

core/ui/.../compose/AdaptiveLayout.kt provides ContentWidth + Modifier.constrainedContentWidth() + ConstrainedTopBar / ConstrainedFloatingAction. These only cap and center a single column on wide screens (Feed 840 / Reading 720 / Form 680 / Auth 480). Used in ~18 files.

Key finding

There is no WindowSizeClass, NavigationRail, NavigationSuiteScaffold, ListDetailPaneScaffold, or any material3.adaptive usage anywhere in the repo. The web build is literally the phone layout, width-capped. That is exactly the “stretched mobile” problem the design sets out to fix.

2.3 Per-screen chrome is duplicated and phone-shaped

Each feature screen owns its own Scaffold + TopAppBar (with a back arrow) + often a FAB. There is no shared shell; navigation chrome is re-implemented per screen. This is fine for a phone stack but is the thing that must move into a single adaptive shell.

4Target adaptive architecture

4.1 Dependencies to add

Compose Multiplatform 1.11 ships the adaptive artifacts as multiplatform libraries. Add to gradle/libs.versions.toml and the core:ui (and/or composeApp) commonMain source set:

# libs.versions.toml
material3AdaptiveNav = "org.jetbrains.compose.material3.adaptive:adaptive-navigation-suite"
material3Adaptive    = "org.jetbrains.compose.material3.adaptive:adaptive"
material3WindowSize  = "org.jetbrains.compose.material3:material3-window-size-class"

These give us currentWindowAdaptiveInfo(), WindowSizeClass, NavigationSuiteScaffold, and ListDetailPaneScaffold / SupportingPaneScaffold. Whether we adopt NavigationSuiteScaffold wholesale or build a custom shell is discussed in §4.3.

4.2 The window-width tier model

Material3’s default WindowWidthSizeClass buckets (Compact <600, Medium 600–840, Expanded ≥840) are coarser than the prototype’s CSS breakpoints. Define a single source of truth that maps width to a WingsLog tier, approximating the design’s px thresholds as dp:

// core/ui/.../compose/LayoutTier.kt  (new)
enum class LayoutTier { COMPACT, MEDIUM, EXPANDED, LARGE }

// Thresholds chosen to mirror the prototype's 720 / 1040 / 1180 CSS breakpoints.
@Composable
fun rememberLayoutTier(): LayoutTier {
  val w = currentWindowAdaptiveInfo().windowSizeClass
      .windowWidthSizeClass /* or raw widthDp */
  return when {
    widthDp < 700  -> LayoutTier.COMPACT    // phone: bottom bar, sheets, 1 col
    widthDp < 1040 -> LayoutTier.MEDIUM     // tablet portrait: icon rail, 1–2 col, drawers
    widthDp < 1240 -> LayoutTier.EXPANDED   // tablet landscape / laptop: sidebar, 2 col, no dash grid
    else           -> LayoutTier.LARGE      // desktop: sidebar + lower dashboard grid, max-width
  }
}
Why a custom tier, not raw WindowSizeClass

The prototype switches the navigation container at 720px and the dashboard lower grid at 1180px — neither lines up with the 600/840 default buckets. Reading raw widthDp from currentWindowAdaptiveInfo() (or constructing a WindowSizeClass with custom breakpoints) keeps the Compose layout faithful to the approved design. The dp values are tunable.

4.3 The shell: AdaptiveAppShell on NavigationSuiteScaffold

Decision D2 — revised

Started on NavigationSuiteScaffold (M1). Because its drawer can’t host the aircraft switcher in the sidebar the way the mock shows, the shell now uses a custom WingsSidebar at EXPANDED/LARGE (brand + switcher + sections + account footer), while MEDIUM keeps the scaffold rail (switcher in the top bar) and COMPACT keeps the scaffold bottom bar + fleet landing.

NavigationSuiteScaffold maps the navigation container by window size automatically (Compact → bottom bar, Medium → navigation rail, Expanded → permanent drawer). We feed it the section items and host the active section in its content slot. The aircraft switcher lives in the top bar (a dropdown) rather than inside the nav container — this avoids fighting the scaffold’s drawer header and works identically across rail and drawer.

// composeApp/.../shell/AdaptiveAppShell.kt  (new)
@Composable
fun AdaptiveAppShell(...) {
  val tier = rememberLayoutTier()

  // COMPACT root is the dedicated fleet landing page (see §6 / D1), not a scoped section.
  // Above phone, the switcher in the top bar selects the ambient aircraft.
  NavigationSuiteScaffold(
    navigationSuiteItems = {
      sections.forEach { s ->
        item(
          selected = s.id == current,
          onClick  = { go(s) },
          icon     = { Icon(s.icon, null) },
          label    = { Text(s.label) },
          badge    = { badgeFor(s)?.let { Badge { Text("$it") } } },
        )
      }
    },
  ) {
    Column(Modifier.fillMaxSize()) {
      WingsTopBar(
        title = …, sub = …,
        switcher = if (tier != COMPACT) { AircraftSwitcher(selected, ::pick) } else null,
        themeToggle = …,
      )
      SectionHost(showDashboardGrid = tier == LARGE) // lower dashboard grid only at LARGE
    }
  }
}

Follow-up (deferred): a custom Row-based shell (WingsSidebar/WingsRail/WingsBottomBar) that reproduces the prototype’s 264px sidebar with the in-rail switcher, section labels, and account footer — only if the default scaffold isn’t good enough. Not in scope for the first pass.

4.4 Detail panes & overlays

Introduce one adaptive container that resolves by tier so call-sites stay tier-agnostic:

// core/ui/.../compose/AdaptiveDetail.kt  (new)
@Composable fun AdaptiveDetailContainer(open: Boolean, onDismiss, content) =
  when (rememberLayoutTier()) {
    COMPACT -> ModalBottomSheet(onDismiss) { content() }   // = today's bottom sheet
    else    -> EndDrawer(width = 460.dp, onDismiss) { content() }  // right-side drawer
  }

Add/edit forms use the same idea: COMPACT → full-screen route (today’s behavior) or bottom sheet; MEDIUM+ → centered Dialog sized like the prototype modal. The form content composable is identical across tiers.

5The switch: single-column → adaptive

How runtime layout selection works, and how we migrate without a big-bang rewrite.

5.1 Runtime mechanism

  1. AppEntry wraps the authenticated graph in AdaptiveAppShell.
  2. The shell reads rememberLayoutTier() once and recomposes when the window resizes (web window drag, tablet rotation, multi-window). Because the tier is derived state, the entire shell reflows — sidebar ⇄ rail ⇄ bottom bar — without navigation events.
  3. Section content is hosted by an inner NavHost whose start destination is the selected section; the outer (auth) NavHost remains for login/onboarding.
  4. Detail/edit invocations call AdaptiveDetailContainer / the adaptive dialog, which pick sheet vs drawer vs dialog by tier — so a row tapped at desktop opens a drawer, the same row at phone opens a sheet.

5.2 Migration strategy — strangler, screen by screen

The content composables already exist; we re-host them. Recommended order keeps the app shippable at every step:

  1. Stand up the shell behind a feature flag (adaptiveShellEnabled in FeatureFlags), defaulting on for web, off for phone, so the existing stack stays intact on mobile until parity is proven.
  2. Re-host sections (Dashboard → Tasks → Logs → Squawks → Settings) one at a time: strip each screen’s own Scaffold/TopAppBar/FAB, expose a pure content composable, and mount it in the shell’s SectionHost.
  3. Convert overlays: wrap existing detail sheets / edit screens in AdaptiveDetailContainer and the adaptive dialog.
  4. Add wide-only affordances: dashboard lower two-column grid, logs table, 2-col grids.
  5. Flip the flag on for all tiers once compact parity is confirmed; delete the legacy graph wiring.

5.3 What stays single-column forever

COMPACT is still the native phone experience — the design is explicit that phone should remain faithful to the current app. So “single-column layout system” is not removed; it becomes the COMPACT branch of the same shell. constrainedContentWidth stays useful for capping content inside a pane at MEDIUM/EXPANDED where there is no second pane.

6Navigation model change (aircraft-scoped)

Biggest behavioral change — and it is tier-dependent

Above phone, aircraft moves from a pushed route argument to an app-level ambient selection: sections render against the selected aircraft and switching swaps content in place. On phone there is no switcher, so COMPACT keeps the current model — a dedicated fleet landing page is the root, and tapping an aircraft enters its scoped sections.

Fleet landing — resolved (D1)

The switcher in the tablet/desktop nav chrome is the fleet overview, so no separate fleet destination is needed there. But because the switcher cannot be shown on phone, COMPACT retains a dedicated fleet landing page — the same aircraft-list entry point the app has today — as the root of the phone experience. Net effect: one fleet-list composable, surfaced as the COMPACT root and (optionally) reachable on wide screens, while the switcher handles aircraft selection above phone.

7Reuse / Refactor / Remove

✓ Reuse (carries over essentially unchanged)

  • Entire domain/data layer — EntityStore, every feature datamanager + Manager, the Firestore sync engine. The redesign does not touch storage.
  • All ViewModels & StateFlow — sections bind to the same VMs.
  • Theme & tokens — WingslogTheme, fonts, palette, status colors, both light/dark. The toggle just surfaces in the top bar.
  • Screen content composables — aircraft overview content, task/log/squawk lists, forms — reused as pane content once their own scaffolding is peeled off.
  • Existing detail bottom sheets — become the COMPACT branch of AdaptiveDetailContainer.
  • ContentWidth / constrainedContentWidth — still used to cap single-pane width at MEDIUM/EXPANDED.

↻ Refactor

  • AppEntry.kt NavHost — the flat 4-graph stack becomes [auth graph] → [AdaptiveAppShell hosting a section NavHost]. Aircraft & global sub-graphs collapse into shell sections.
  • Aircraft identity plumbing — from route arg to ambient selectedAircraftId; Screen.MaintenanceOverview and friends are reframed.
  • Per-screen chrome — remove each screen’s Scaffold/ TopAppBar/back-arrow/FAB; expose content-only composables. Primary actions move to the top bar / toolbar (FAB → toolbar button on wide; FAB may remain at COMPACT).
  • Dashboard — keep aircraft data and maintenance summary full-width, then split lower content into Recent Activity left and Next Due/Open Squawks right at LARGE.
  • Logs — add a table layout for MEDIUM+; keep cards for COMPACT.
  • AircraftOverviewScreen internal tabs — Overview/Tasks/Logs tabs are promoted to shell-level sections; the overview’s stat blocks become the 2-column grid.

✕ Remove / retire

  • “Stretched mobile” assumption — single-column-only as the web layout.
  • Back-button-driven aircraft navigation on large screens (replaced by the switcher; back stack still applies to overlays/forms).
  • Export as a top-level destination — per chat6 it lives inside Settings only.
  • Duplicated per-screen top bars on wide screens — superseded by the shell top bar (the composables themselves are refactored, not all deleted, since COMPACT may still use a compact top).
  • FAB on wide screens — replaced by toolbar/primary buttons.
  • (Decision) the standalone fleet Dashboard list as the only root — see §6/§11.

9Per-screen plan

Dashboard (per-aircraft)

Hero (fleet/aircraft status) + status tiles (overdue / due-soon / airworthy) + aircraft card(s). LARGE adds the lower dashboard grid; EXPANDED and below stack those cards under the main column.

Tasks

Segmented control (Due / History) + card grid (2-col at EXPANDED+, 1-col below). Card → detail in AdaptiveDetailContainer. Add/edit via adaptive dialog (tabs: Identity / Details / Schedule / Overrides). Overdue badge feeds the sidebar/bottom-bar + switcher.

Logs

Toolbar (search + type chips). LogsTable at MEDIUM+ with sticky header and row→drawer; cards at COMPACT. Add-log via adaptive dialog/sheet.

Squawks

2-col card grid (grounding/monitor/minor/closed). “Closed” rolls up deferred + resolved (chat6). Card → drawer; add/edit via adaptive container. Open-count badge.

Settings (global)

Profile card + appearance (theme switch) + time source + notifications + technicians + sync + Export (entry point lives here). Reading-width single column; no second pane. Current implementation gap: ShellSection.SETTINGS still renders SettingsSectionPlaceholder, whose button pushes the legacy standalone SettingsScreen. M6 replaces that escape hatch with native settings content inside the adaptive shell.

Export

Config column + sticky live summary at MEDIUM+; stacked at COMPACT. Works as a page or a modal.
