Hopply — Promo Site Design

Status: Draft — not yet implemented

Single-page static marketing site hosted at hopply.dev (Firebase Hosting). The Hopply web app lives at app.hopply.dev on a separate Firebase Hosting site. This document covers page structure, visual design, SEO requirements, and the exact content assets the author must provide before implementation begins.

Related docs: web_target_expansion_plan.md · PRD.md

1. Goals

  • Drive organic search traffic for queries like "aviation maintenance logbook app", "aircraft maintenance tracking app", "FAA maintenance log app".
  • Funnel visitors to the correct app store (iOS / Android) or the web app.
  • Load fast — no JavaScript required. Pure HTML + CSS, zero frameworks.
  • Render correctly on mobile (the primary discovery device for app searches).
  • Support social sharing previews (Open Graph, Twitter Card).

2. Implementation Stack

  • HTML5 + Tailwind CSS via CDN — single index.html, no build step.
  • Space Grotesk loaded from Google Fonts (display + headlines only).
  • JetBrains Mono loaded from Google Fonts (tail number / technical data accents).
  • No JavaScript, no analytics scripts inline — analytics tag can be appended later.
  • Source location: site/index.html at the repo root, alongside webApp/.

5. SEO Requirements

All items below belong in the <head> of site/index.html or as in-page structured data. They are the minimum required for effective indexing and social sharing.

5E. Favicon & App Icons

Required assets:
  • site/favicon.ico — 32×32 or multi-size ICO.
  • site/favicon-192.png — 192×192 PNG (Android home screen shortcut).
  • site/apple-touch-icon.png — 180×180 PNG (iOS "Add to Home Screen").
  • site/icon.png — 512×512 PNG (used in structured data + OG fallback).
All four can be exported from the same master app icon artwork.

5F. On-Page SEO

  • One <h1> per page — the display headline in the Hero section. Must include a primary keyword (e.g. "aviation maintenance logbook").
  • Section headings use <h2>. Feature card titles use <h3>.
  • Every <img> has a descriptive alt attribute (e.g. alt="Hopply iOS app showing aircraft inspection due dates").
  • App Store and Play Store links use rel="noopener" and open in a new tab.
  • Page is fully readable with CSS disabled (crawler fallback).

5G. Performance (Core Web Vitals)

Core Web Vitals are a direct Google ranking signal. The static HTML approach already wins here, but a few specifics matter:

  • Serve screenshot images as WebP with a JPEG fallback (<picture> element). WebP is ~30% smaller.
  • Add width and height attributes to every <img> to prevent layout shift (CLS).
  • Add loading="lazy" to below-the-fold screenshots.
  • The hero screenshot (above the fold) should not be lazy-loaded — add <link rel="preload"> for it instead.
  • Google Fonts: use display=swap in the font URL and preconnect to fonts.gstatic.com.

6. Asset Checklist

Everything the author must produce before the page can be built.

  • App icon — 512×512 PNG, no rounded corners
  • favicon.ico (32×32)
  • favicon-192.png (192×192)
  • apple-touch-icon.png (180×180)
  • Open Graph image — 1200×630 PNG or WebP (see §5B)
  • Hero screenshot with device frame — at least 1080px tall
  • In-app screenshot: Fleet Dashboard (aircraft list, DUE SOON chip visible)
  • In-app screenshot: Aircraft Overview (tail number + maintenance summary)
  • In-app screenshot: Maintenance Log entry
  • In-app screenshot: Tasks list (recommended)
  • In-app screenshot: Squawks list (recommended)
  • App Store URL
  • Google Play URL
  • Privacy Policy URL
  • Page title (~60 chars)
  • Meta description (~150 chars)
  • 1–2 real user quotes for social proof section (optional)
Screenshots are the highest-ROI investment on this checklist. Google indexes image content and uses app screenshots in rich result cards. Poor screenshots (small, blurry, showing empty states or placeholder data) will undermine the page even if every meta tag is perfect. Use a real device or high-fidelity simulator with realistic fleet data populated.
