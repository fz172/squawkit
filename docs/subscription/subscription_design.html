Subscription — SquawkIt Pro

Engineering design for the subscription feature. Companion to the PRD: how the account-level entitlement is stored, synced, and gated on; the Feature Lab → Developer Options refactor; the server pipeline abstracted over an undecided billing provider; storage-usage accounting; and a phased build roadmap.

Implementation Status Proposed. Not yet built. The billing integration (native vs. aggregator) is an explicit blackbox — everything here is written to be agnostic to it, and the purchase/paywall UI is out of scope until that call is made (PRD §14). This note will track what actually ships.

1. Overview & Principles

Three architectural commitments drive the whole design:

  • Entitlement is server-authoritative and account-scoped. A trusted Cloud Function is the only writer; the client reads a synced, read-only copy. This is what makes "pay once, premium everywhere" structural rather than client logic (PRD §6).
  • The client gates on a tier enum, resolved locally. A single SubscriptionManager exposes an effective Subscription.Status (the proto enum — no forked types, §4) that features compare against a per-feature minimum tier. It works offline off the last synced entitlement.
  • Gate as promo, don't hide. Gated affordances stay visible and open a reusable ProUpsellSheet (PRD §9).
  • Rolled out through AppCapability, default-open. A new isSubscriptionSupported build gate (like isAircraftSharingSupported) stages the launch: on in dev/dogfood, off in the shipping release until GA. While it is off there is no paywall at all — every premium feature is treated as available (no gating), and the Subscription entry/page is hidden. Off means "unlocked for everyone," not "restricted."

The design leans on existing infrastructure where it can: the local-first EntityStore caches the entitlement for offline reads, and the storage-usage number is summed by extending the scheduled storage sweep. The one deliberate deviation is the entitlement's remote path: because it must be un-forgeable it cannot ride the client-writable user tree (§3), so a small dedicated listener mirrors a top-level, function-only doc into that local cache.

3. Data Model & Storage

The entitlement record

One document per account at the top-level, function-only path subscriptions/{uid} (see "why not the user tree" below), written only by the server. It holds both the entitlement and the storage-usage figure (two writers touching disjoint fields — the billing pipeline and the sweep — via Firestore field merges):

// core/model … settings/Subscription.proto  (wire name: "subscription")
message Subscription {
  Status  status              = 1;  // resolved tier the server grants
  Lifecycle lifecycle         = 2;  // billing state (see below)
  int64   member_since_millis = 3;  // earliest purchase/trial start
  int64   current_period_end_millis = 4;
  bool    will_renew          = 5;
  Source  source              = 6;  // STORE_PURCHASE | SERVER_GRANT
  string  origin_platform     = 7;  // "ios" | "android" | "web" — informational
  int64   storage_bytes_used  = 8;  // written by the storage sweep (§7)
  int64   storage_quota_bytes = 9;  // 0 = unlimited (v1); reserved for future tiers

  enum Status    { STATUS_FREE = 0; STATUS_PRO = 1; }         // ordered, extensible
  enum Lifecycle { NONE = 0; TRIALING = 1; ACTIVE = 2; CANCELED = 3; GRACE = 4; EXPIRED = 5; }
  enum Source    { SOURCE_UNSPECIFIED = 0; STORE_PURCHASE = 1; SERVER_GRANT = 2; }
}

The proto enums carry a 0-valued default, so an absent doc reads as Subscription.Status.STATUS_FREE. Because the app has not launched, we can define wire names and fields freely — no migration constraints.

CollectionKind + local cache

The client caches the entitlement in the local store like any other entity, so the manager reads it offline through an EntityStore<Subscription>. The local cache is keyed at userRoot(uid); only the remote source differs (a dedicated listener, below — not the generic sync).

data object Subscription : CollectionKind {
  override val wireName = "subscription"
  override val schemaName = "settings.Subscription"
}

Server-authoritative — why it lives outside the user tree

The entitlement is a paywall, so it must be un-forgeable: only the server may write it. That rules out putting it under the user's own tree. The user tree is governed by one broad rule —

match /users/{userId}/{document=**} { allow read, write: if request.auth.uid == userId }

— and Firestore rules are additive (OR): a more-specific allow write: if false on a subpath cannot revoke the broad allow write. So anything under users/{uid}/ is client-writable, and a user could forge STATUS_PRO by writing the doc straight to Firestore with their own token. (This is also why SharedAircraftRef, which lives under the user tree, is not a write-denied precedent — it relies on that same broad allow.) The entitlement therefore lives at a top-level path the broad rule doesn't cover, where writes are genuinely denied:

match /subscriptions/{uid} {
  allow read:  if request.auth.uid == uid;
  allow write: if false;   // Admin SDK (Cloud Functions) only
}

Read path (one small piece of new machinery). Because that doc is outside the user tree, the generic hydration doesn't fetch it. A dedicated SubscriptionSyncListener (bound in the sync/subscription layer) subscribes to subscriptions/{uid} while signed in and mirrors it into the local EntityStore (CollectionKind.Subscription). The listener is the only local writer of that row, so it is never marked dirty and never pushed; SubscriptionManager observes the local cache exactly as designed. A single-doc listener, not a new bidirectional collection.

4. SubscriptionManager & Gating

The single reactive gate. Mirrors FeatureLabManager's shape (auth-scoped flatMapLatest over the store), adding lifecycle resolution and the developer override.

No generic PremiumFeature enum. The gated things are not homogeneous, and there is no single existing enable/disable boundary to key an enum on: two are booleans that already have a flag to replace (attachment upload, email export), one is a boolean gate that does not exist yet (hosting a share), and the aircraft limit is not boolean at all — it is a quantity. So the manager exposes explicit capability accessors (one per real gate site) and a separate limit, rather than a uniform hasAccess(feature) that would imply an enforcement hook the codebase doesn't have.

Reuse the proto as the model — don't fork enums or classes. The Subscription proto (core:model, P1b) already defines Status (ordered STATUS_FREE < STATUS_PRO), Lifecycle, and Source, and the Cloud Functions share the exact same enums via their generated TS. Re-declaring parallel Kotlin enums (or a hand-written Entitlement class) is pure drift risk, and it cuts against the repo convention of using proto types as domain models (Aircraft, Squawk, …). So the tier is Subscription.Status and the entitlement is the Subscription proto itself; feature/subscription/model holds only the logic over it.

// feature/subscription/model — no new types, just the resolution over the proto:
fun Subscription.effectiveStatusAt(nowMillis: Long): Subscription.Status

interface SubscriptionManager {
  fun status(): Flow<Subscription.Status>           // status page + comparison table
  fun entitlement(): Flow<Subscription>             // the proto: memberSince, periodEnd, usage

  // Boolean capabilities — one gate site each (§6):
  fun canUploadAttachments(): Flow<Boolean>         // replaces attachmentUploadEnabled
  fun canEmailExports(): Flow<Boolean>              // replaces exportEmailDeliveryEnabled
  fun canHostShare(): Flow<Boolean>                 // NEW gate at the Share entry

  // Quantity limit — enforced against current usage, not a yes/no:
  fun aircraftLimit(): Flow<Int?>                   // null = unlimited; FREE = 1
}

Rollout gate first. Before any entitlement logic runs, the manager checks AppCapability.isSubscriptionSupported. While the capability is off, gating is disabled: every capability accessor emits true and aircraftLimit() emits null (unlimited). No entitlement doc even needs to exist. status() is irrelevant in this state because its only consumer, the Subscription page, is hidden (§8). When on, effective status = lifecycle resolved to a tier (TRIALING/ACTIVE/CANCELED-before-end/GRACE → the granted tier; NONE/EXPIRED → FREE), then the developer override OR-ed on top. A private gate(minimum) helper centralizes the capability → minimum-tier mapping:

private fun gate(minimum: Subscription.Status): Flow<Boolean> =
  if (!appCapability.isSubscriptionSupported) flowOf(true)          // no paywall yet → open
  else status().map { it >= minimum }                              // proto enums order by declaration

override fun canUploadAttachments() = gate(STATUS_PRO)
override fun canEmailExports()      = gate(STATUS_PRO)
override fun canHostShare()         = gate(STATUS_PRO)
override fun aircraftLimit(): Flow<Int?> =
  if (!appCapability.isSubscriptionSupported) flowOf(null)          // unlimited while off
  else status().map { if (it >= STATUS_PRO) null else FREE_AIRCRAFT_LIMIT } // = 1

// status(), capability on:
val effective = when {
  appCapability.isDeveloperOptionsSupported && devOptions.forceStatus != null -> devOptions.forceStatus
  else -> subscription.effectiveStatusAt(nowMillis)   // lifecycle + current_period_end vs now
}

The override is honored only in developer builds (guarded by AppCapability), so a release build can never be forced premium even if the field were somehow set. If the comparison page (§3 of the PRD) wants a data-driven "what each tier includes," that is static presentation data, kept separate from these enforcement accessors — not the same enum.

Add-aircraft is a limit, not a capability Free allows one aircraft, not zero — so it is a quantity (aircraftLimit(): null=unlimited, FREE=1), not a boolean feature. The add-aircraft path compares aircraftLimit() against the owned count from FleetManager.observeFleetDashboard(), filtered to owned aircraft — shared-in aircraft (SharedAircraftRef) never count (PRD R4). Future storage quota is the same shape, which is exactly why it belongs with limits rather than a fake yes/no. Bypassed while isSubscriptionSupported is off (unlimited for everyone).

5. Feature Lab → Developer Options Refactor

Ships with the gating work, because the two flags can't be removed until their features re-gate on the subscription. Pre-launch, so proto/wire renames are free.

  • Rename module feature/featurelab → feature/developeroptions; FeatureLabManager → DeveloperOptionsManager; proto FeatureLabSettings → DeveloperSettings; CollectionKind.FeatureLab → DeveloperOptions.
  • Remove attachmentUploadEnabled and exportEmailDeliveryEnabled. Add a nullable force-status dev override typed as Subscription.Status? (stored in the developer-settings proto with an added UNSET sentinel so "off" is distinguishable from FREE).
  • FeatureLabScreen → DeveloperOptionsScreen: a segmented Force subscription status control (Off / Free / Pro) + the existing Generate fake data extra (StressTestFeatureLabExtra, renamed). Keep the AppCapability.isFeatureLabSupported gate (rename → isDeveloperOptionsSupported).
  • Settings row + Screen.FeatureLab → Screen.DeveloperOptions; string "Feature Lab" → "Developer Options".

DeveloperOptionsManager keeps the same read/write plumbing (it's a locally-written, per-account doc) — only its contents change. SubscriptionManagerImpl depends on it to read forceStatus.

7. Backend — Entitlement Pipeline & Usage

Writer, abstracted over the billing blackbox

One internal contract, one writer; the billing provider is a pluggable source that normalizes into it. This lets us build and test the whole client + writer before choosing native vs. aggregator.

// normalized, provider-agnostic
interface EntitlementSource {           // impl: RevenueCatWebhook | AppleAsn | PlayRtdn | Stripe
  // maps a raw provider event → NormalizedEntitlement { uid, status, lifecycle,
  //   memberSince, currentPeriodEnd, willRenew, source, originPlatform }
}
// the only writer of subscriptions/{uid} (entitlement fields), idempotent
function applyEntitlement(update: NormalizedEntitlement): Promise<void>
  • Ingest: an HTTPS webhook (aggregator) or store notifications (Apple ASSN v2, Play RTDN via Pub/Sub, Stripe) — decided later. Each validates the receipt/signature server-side and hands a NormalizedEntitlement to applyEntitlement.
  • Idempotency: keyed on the provider's transaction/event id so retries and re-delivery don't double-apply.
  • Promos (SERVER_GRANT): an admin-only callable (or console script) writes a comp'd entitlement with an end date and no store purchase — no billing provider involved.
  • Client never self-grants from a local receipt; it only reads the synced result.
v1 scope of the pipeline Build the applyEntitlement writer, the security rules, the promo callable, and a stub ingest that the Force-subscription dev toggle and manual promo grants exercise end-to-end. The real provider webhook + purchase/paywall UI land after the billing decision.

Storage-usage sweep

Per PRD §6, usage is summed server-side. Extend the scheduled storage sweep: for each account, one paginated LIST on the single prefix users/{uid}/ (where all blobs live), sum file.metadata.size, and write storage_bytes_used onto the subscriptions/{uid} doc via field merge. Authoritative, drift-free, no upload-path hook. Real-time incremental counting (onObjectFinalized/onObjectDeleted, reconciled by the sweep) is deferred to hard-quota enforcement. Because blobs live under the owner/host uid, shared-aircraft bytes accrue to the host.

9. Testing

  • SubscriptionManagerImpl: lifecycle→status resolution (canceled-before-end = PRO; expired = FREE; grace = PRO), dev-override honored only when isDeveloperOptionsSupported, offline reads from cache. (MockK + Truth + coroutines-test, per the repo harness.)
  • Add-aircraft composite gate: 0/1/2 owned, ×{FREE,PRO}, shared-in excluded.
  • Each consumer VM: emits locked vs. granted from a faked manager.
  • Functions (vitest + emulator): applyEntitlement idempotency; security rules deny client writes / allow owner reads; sweep sums a known bucket layout; promo grant expiry.

11. Open Questions

  • Billing provider (PRD §14) — gates P7 only. Everything above is written against the normalized EntitlementSource so the choice is isolated.
  • Usage cadence — how often the sweep recomputes usage (daily vs. hourly); display freshness follows it.
  • Guest paywall routing — the exact hop from a locked affordance → account upgrade → purchase (guests can't subscribe; PRD §14).
