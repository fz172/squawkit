Passwordless Email-Link Sign-In — Technical Design

A third sign-in option — Log in with email — that places an email-link (passwordless) button between Log in with Apple and Continue Anonymously on every platform. The user types an email, receives a one-time sign-in link, and taps it to complete authentication. This doc covers the shared AuthManager contract, the two-leg send/complete flow, and the per-platform link-handling that makes the tap land back in the app. Sibling to the Guest → Account Upgrade design.

1. Goal & Scope

Firebase's Email/Password provider with Email link (passwordless sign-in) is already enabled in the console. This design wires that provider into SquawkIt as a first-class sign-in method, ranked third in the login UI:

  1. Log in with Google (existing)
  2. Log in with Apple (existing button; backend still stubbed)
  3. Log in with email — this design
  4. Continue Anonymously (existing; hidden on web)

In scope: the send-link + complete-link flow on Android, iOS, and web; the shared AuthManager surface; UI for email entry and the "check your inbox" state; and a feature-flag gate. Out of scope: linking an email credential onto an existing anonymous guest (that belongs in the account-upgrade flow and is called out as a follow-up in §8); custom email templates beyond the Firebase console defaults.

2. How email-link sign-in works

Unlike Google/Apple (a single synchronous popup/sheet that returns a credential), email-link is a two-leg flow split across apps and possibly devices:

LEG 1 — SEND LEG 2 — COMPLETE ┌─────────────┐ ┌──────────────────┐ │ Login screen│ │ Email client │ │ enter email │ │ taps sign-in link│ └──────┬──────┘ └────────┬─────────┘ │ sendSignInLinkToEmail(email, acs) │ deep link / URL ▼ ▼ ┌─────────────┐ persist email ┌──────────────────┐ │ Firebase │ ───────────────► │ SquawkIt app │ │ sends email │ (local store) │ isSignInWithEmail│ └─────────────┘ │ Link? → signIn │ └────────┬─────────┘ │ FirebaseUser ▼ → onLoginSuccess()

The four primitives, all available on GitLive firebase-auth 2.4.0 across Android / iOS / JS targets (verified in the published klibs):

PrimitiveLegPurpose
FirebaseAuth.sendSignInLinkToEmail(email, ActionCodeSettings)SendEmails a one-time link. ActionCodeSettings.handleCodeInApp = true is required for email-link.
FirebaseAuth.isSignInWithEmailLink(link)CompleteCheap check: is this inbound URL an email sign-in link?
FirebaseAuth.signInWithEmailLink(email, link)CompleteExchanges the link (+ the email we stashed) for a FirebaseUser.
EmailAuthProvider.credentialWithLink(email, link)UpgradeBuilds an AuthCredential for linkWithCredential — used only by the future guest-upgrade path (§8).
Why we must persist the email locally signInWithEmailLink needs the email address that the link was issued for. When the link is opened on the same device we read it back from local storage. When it's opened on a different device (or the store was cleared), we have no stored email and must prompt the user to re-enter it before completing — this is a required branch, not an edge case.

3. The continue URL (ActionCodeSettings)

Every platform sends the link with the same hosted continue URL so the experience and the deep-link verification files are shared. We already own squawkit.fanfly.dev (the Firebase Auth authDomain and the Hosting target), so the link returns there:

// core:auth — shared ActionCodeSettings builder
val actionCodeSettings = ActionCodeSettings(
  url = "https://squawkit.fanfly.dev/finishSignIn",  // must be allow-listed in Auth → Settings → Authorized domains
  androidPackageName = AndroidPackageName(
    packageName = "dev.fanfly.wingslog",
    installIfNotAvailable = true,
    minimumVersion = null,
  ),
  iOSBundleId = "dev.fanfly.wingslog",
  canHandleCodeInApp = true,            // REQUIRED for email-link
  dynamicLinkDomain = null,             // see callout — we do NOT use Dynamic Links
)
Firebase Dynamic Links is shut down — use App Links / Universal Links Dynamic Links reached end-of-life in 2025, so dynamicLinkDomain is left null. On mobile the continue URL is opened as a plain HTTPS link and routed into the app by Android App Links (verified assetlinks.json) and iOS Universal Links (an apple-app-site-association file). If verification is missing, the OS opens the URL in the browser instead of the app — which on web still completes sign-in, so the browser is the graceful fallback for mobile too. Both verification files are served from squawkit.fanfly.dev via Firebase Hosting (§7).

4. Shared contract: AuthManager & LoginViewModel

Two new suspend functions on the common AuthManager interface — one per leg. They join the existing signInWithGoogle() / signInAnonymously() family and keep the same "return the user or null" convention.

// core/auth/src/commonMain/.../AuthManager.kt
interface AuthManager {
  // …existing members…

  /** LEG 1 — emails a one-time sign-in link and stashes [email] for completion. */
  suspend fun sendSignInLink(email: String): SendLinkResult

  /**
   * LEG 2 — if [link] is a sign-in link, completes auth using [fallbackEmail] when no
   * email was stashed locally (cross-device case). Returns null when [link] is not a
   * sign-in link, so callers can ignore unrelated deep links.
   */
  suspend fun completeSignInLink(link: String, fallbackEmail: String? = null): FirebaseUser?
}

SendLinkResult is a small sealed type (Sent(email) / InvalidEmail / Failed(message)) so the UI can distinguish a validation problem from a network failure. The stashed email lives in the same local key-value store the rest of core:auth / onboarding already uses (the implementation reads/writes a single pending_signin_email key).

LoginViewModel gains thin pass-throughs, matching its existing style:

// feature/login/.../LoginViewModel.kt
suspend fun sendEmailLink(email: String) = authManager.sendSignInLink(email)
suspend fun completeEmailLink(link: String, fallbackEmail: String? = null) =
    authManager.completeSignInLink(link, fallbackEmail)

5. UI changes

The email button is added to both sign-in surfaces (the native button column and the web promo card), but it does not reveal an inline form. Instead it navigates to a single dedicated Email Sign-In page that is shared verbatim by Android, iOS, and web. That page wears the same header as the native login screen (bobbing-plane mark + SquawkIt. wordmark + mission statement). On web this deliberately replaces the marketing landing — once the user commits to logging in, the promo content is gone and they see the same focused login UI as on mobile.

One screen, three platforms The Email Sign-In page is a new shared composable, EmailSignInScreen, in feature/login commonMain. There is no web-specific variant and no web-specific view model — web reuses the same EmailSignInScreen and the same LoginViewModel as mobile. This is the simplification driver: the promo page and the login page are now cleanly separate concerns.

5.1 Entry points (the third button)

On native (LoginScreen.kt) a third button is inserted directly after Apple and before the if (isAnonymousLoginSupported) block, so ranking is correct everywhere. Same 54 dp height, Spacing.buttonCornerRadius, and disabled-while-signing-in treatment as the others; styled as an outlined/tonal button (not a brand color) with a mail glyph, reading as the "neutral third option" beneath the two branded providers.

Button(Google)        // existing
Spacer(medium)
Button(Apple)         // existing
Spacer(medium)
Button(Email)         // NEW — onChooseEmail() → navigate to EmailSignInScreen
if (isAnonymousLoginSupported) {
  Spacer(medium)
  OutlinedButton(Anonymous)   // existing
}

On web the promo card's LoginCard (in WebLoginLandingScreen.kt) gains the matching third AuthButton ("Log in with email") below Apple. Its onClick raises the same onChooseEmail — it does not render any in-card state.

5.2 Navigation: who hosts the shared page

To make the page truly shared (and to let it replace the promo on web), the Options ↔ Email toggle is hoisted into AuthFlow rather than living inside each surface. The loginContent slot gains an onChooseEmail callback, and the Login step renders either the platform options view or the shared email page:

// feature/login/.../AuthFlow.kt — Login step becomes a 2-screen sub-flow
enum class AuthStep { Login, EmailSignIn, NameEntry, Welcome }

when (step) {
  Login -> loginContent(
    onLoginSuccess = { /* existing post-login routing */ },
    onChooseEmail  = { step = AuthStep.EmailSignIn },   // NEW
  )
  EmailSignIn -> EmailSignInScreen(
    onBack         = { step = AuthStep.Login },
    onLoginSuccess = { /* same routing as Login */ },
  )
  NameEntry -> …
  Welcome   -> …
}

So on native, tapping Email leaves LoginScreen for EmailSignInScreen; on web, tapping Email leaves WebLoginLandingScreen (promo) for the very same EmailSignInScreen. onBack returns to whichever options surface the platform supplied.

5.3 The shared Email Sign-In page

To guarantee an identical header, the existing header block in LoginScreen (the animated plane icon, the SquawkIt. wordmark with the blue dot, and the mission statement) is extracted into a small shared LoginHeader composable and reused by both LoginScreen and EmailSignInScreen. Below the header, EmailSignInScreen renders one of two states:

  • Email entry: a single OutlinedTextField (email keyboard + autofill hint), a primary Send link button, and a back affordance to the options screen. On Sent it advances to the "link sent" state; on InvalidEmail it shows inline field-error text; on Failed it shows an error line below the field.
  • Link sent ("check your inbox"): confirmation copy naming the address, the Resend action with its 60-second countdown (§5.4), and Use a different email to return to the entry state. The screen keeps observing auth state so that when the link is opened on the same device and the app resumes, onLoginSuccess() fires automatically.

The legal disclaimer and privacy link from the login screen are kept at the bottom so the page is self-contained on every platform.

5.4 Resend cooldown (60-second countdown)

After a link is sent, Resend is immediately disabled for 60 seconds to prevent spamming the user's inbox and to stay under Firebase's send rate limits. The button greys out (its enabled flag is false, so it picks up the standard disabled container/content colors) and its label shows a live countdown; at zero it re-enables and reverts to the plain label.

// EmailSignInScreen — cooldown state
var secondsLeft by remember { mutableStateOf(0) }   // 0 == enabled

LaunchedEffect(secondsLeft) {
  if (secondsLeft > 0) { delay(1_000); secondsLeft-- }
}

fun onSent() { secondsLeft = 60 }        // called after a successful send AND on first arrival

Button(
  enabled = secondsLeft == 0 && !isSending,   // greyed out during cooldown
  onClick = { scope.launch { send(); onSent() } },
) {
  Text(if (secondsLeft == 0) resendLabel       // "Resend link"
       else resendInLabel(secondsLeft))            // "Resend in 0:47"
}
The countdown starts on arrival, not only on tap Entering the "link sent" state seeds secondsLeft = 60, because a link was just sent by the entry step. So the cooldown is in effect the moment the user lands on the confirmation screen, and each subsequent Resend restarts it. Use a different email clears it. The single LaunchedEffect(secondsLeft) re-launches each tick, giving a 1 Hz countdown without a manual timer.

5.5 New string resources

Added to feature/login/.../values/strings.xml: sign_in_with_email ("Log in with email"), email_entry_title, email_entry_hint, email_send_link, email_link_sent_title, email_link_sent_body (takes the address), email_resend ("Resend link"), email_resend_in (takes m:ss, e.g. "Resend in 0:47"), email_use_different, email_invalid, and email_link_error. The existing login_prompt copy is left as-is.

Web has no "Continue Anonymously," so email is the only non-OAuth path there On web isAnonymousLoginSupported == false, so the email button is especially valuable: it's the route for users without (or unwilling to use) a Google or Apple account. It is the last option in the promo card, and choosing it takes them to the same shared page as mobile.

6. Per-platform link handling

Leg 1 (send) is identical everywhere — it's pure GitLive SDK. The platform differences are entirely in Leg 2: how the tapped link re-enters the app and reaches completeSignInLink(). The pattern mirrors how Google sign-in is already bridged per platform.

6.1 Android

AuthManagerImpl(android) implements both legs directly against the injected FirebaseAuth (no Credential Manager needed — this is not a Google credential). Link delivery uses App Links:

  • Add a verified VIEW intent-filter to MainActivity in app/src/main/AndroidManifest.xml for https://squawkit.fanfly.dev/finishSignIn with android:autoVerify="true". (The current manifest has only the LAUNCHER filter; the genericidp/recaptcha filters seen in the merged manifest are injected by the Firebase Auth SDK and are unrelated.)
  • MainActivity reads intent?.data on create and in onNewIntent, hands the URL to the shared flow, which calls loginViewModel.completeEmailLink(url, fallbackEmail). A successful result drives the same onLoginSuccess() the buttons use.
  • Host /.well-known/assetlinks.json with the app's SHA-256 signing-cert fingerprints (debug, dogfood, release) so the OS opens the link in-app rather than the browser.

6.2 iOS

Mirrors the existing IosGoogleSignInBridge / MainEntry seam. AuthManagerImpl(ios) calls the GitLive SDK for both legs; the inbound URL is delivered from Swift:

  • Add an Associated Domains entitlement (applinks:squawkit.fanfly.dev) and host /.well-known/apple-app-site-association for the app's Team ID + bundle dev.fanfly.wingslog.
  • Extend the existing .onOpenURL { } in iosApp.swift (and add the continue userActivity handler for Universal Links) to forward the URL: first give it to GIDSignIn (current behavior), and if that returns false, call a new MainEntry.shared.handleIncomingUrl(url) — symmetric to installGoogleSignInHandler/completeGoogleSignIn.
  • handleIncomingUrl routes into shared Kotlin, which calls completeSignInLink and resumes the login flow.

6.3 Web

Simplest case — the link is a navigation to our own Hosting domain, so the URL arrives as window.location.href. The **→/index.html rewrite already in firebase.json means /finishSignIn?... loads the SPA. On startup WebApp passes window.location.href to completeSignInLink; if isSignInWithEmailLink is true it completes sign-in, then strips the auth params from the URL (history replaceState) so a reload doesn't re-trigger. GitLive's JS target wraps the same modular SDK already used for signInWithPopup, so no extra external bindings are required.

8. Interaction with the guest-upgrade flow (follow-up)

A signed-out user picking email gets a brand-new account — straightforward. The richer case is an anonymous guest choosing email to make their account permanent without losing local data. That reuses the UID-preserving machinery in the account-upgrade design: build the credential with EmailAuthProvider.credentialWithLink(email, link) and call linkWithCredential instead of signInWithEmailLink, with the same CredentialInUse → merge fallback. This is intentionally deferred: the upgrade flow today is Google-only and gated behind a Feature Lab flag, and adding the email credential there is a clean, separable follow-up.

9. Feature flag & rollout

Gate the email button behind a FeatureFlags entry (e.g. emailLinkSignInEnabled) via FeatureLabManager, matching how attachmentUploadEnabled and the upgrade flow are gated. This lets us ship the UI dark, validate the App-Links/Universal-Links verification in dogfood, and enable per-environment. The button simply isn't composed when the flag is off, so ranking degrades gracefully back to Google / Apple / Anonymous.

10. Risks & edge cases

CaseHandling
Link opened on a different deviceNo stashed email → UI prompts for the address, passed as fallbackEmail to completeSignInLink.
App Links / Universal Links not yet verifiedOS opens the URL in the browser; the web SPA completes sign-in. Verified files make it open in-app; both succeed.
Expired or already-used linkGitLive throws → surface email_link_error with a Resend action.
Reload after web completionreplaceState strips auth params so the link isn't re-consumed.
Email enumeration / abuseFirebase rate-limits link sends; the UI enforces a 60-second Resend cooldown (§5.4). No "account exists" signal is shown.
Wrong email typedLink still sends to that address; it just never arrives for the user. Use a different email lets them retry.
