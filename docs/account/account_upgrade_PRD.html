Status: Proposed PRD Auth · Local-first · Sync

Guest → Account Upgrade

Let an anonymous (guest) user create a permanent account without losing any of the records they have already entered, syncing the records on device to a real, multi-device account.

Decisions locked for this PRD Providers: Google (Android) + Apple (iOS), App Store compliant. Collision behavior: merge guest data into the existing account. Entry point: Settings profile area only. Rollout: gated behind a Feature Lab flag, off by default. See the companion Design Doc for the technical approach.

1. Problem & Background

SquawkIt lets people start immediately as a guest ("Continue without account" on the login screen, backed by Firebase anonymous auth). Guests get the full local-first experience: aircraft, maintenance logs, tasks, squawks, and attachments are written to the on-device SQLDelight store. The catch is severe and invisible:

  • Guest data is local-only and never synced to an account. The sync engine is idle for anonymous users (user == null || user.isAnonymous → no hydration, no push). Nothing leaves the device.
  • Logging out erases everything, permanently. We just shipped a warning dialog for exactly this, but a warning is a tombstone, not a solution.
  • There is no non-destructive path to a real account today. The login screen's "Sign in with Google" calls signInWithCredential, which swaps the anonymous session for a brand-new UID. Because every local record is keyed by userRoot(uid), switching UID orphans all guest data.

The user most exposed is precisely the one we most want to keep: a pilot or mechanic who trusted the app enough to log real, safety-critical records as a guest. Device loss, a reinstall, or a mistaken logout wipes weeks of work. That directly violates the product's first principle, Dependability First.

2. Goals & Non-Goals

Goals

  • A guest can convert to a permanent Google/Apple account without re-entering or losing any data.
  • The upgrade is zero-friction: one provider tap, no data migration the user has to think about.
  • After upgrade, all existing on-device records are synced to the account and available on the user's other devices.
  • If the chosen account already has SquawkIt data, the guest's on-device records are merged in, not dropped.
  • The flow never destroys data on cancel, error, or interruption.

Non-Goals

  • Downgrade (permanent account → guest).
  • Email/password or phone providers (Google + Apple only for v1).
  • A manual, record-by-record conflict-resolution UI for duplicates created on merge (v1 accepts additive union; see Risks).
  • Linking a second provider to an already-permanent account (account management is out of scope).
  • Changing the existing first-run login screen's two-button layout.

  • Guardrail: records-lost-on-upgrade must be effectively zero; merge collision rate and post-merge duplicate rate are watched.
  • Counter-metric: upgrade attempts that fail or are abandoned mid-flow (should stay low and never lose data).

6. UX & Flow

Entry: Settings profile

For an anonymous user, the Settings profile card presents a guest identity plus a prominent call to action and a quiet reassurance of why:

Profile card (guest state)

Guest · Your records are saved only on this device.
Create account to sync

Happy path (clean link)

  1. Tap Create account to sync → the platform sign-in sheet opens directly (Google on Android, Apple on iOS). No interstitial promo or benefits screen.
  2. Progress: "Syncing your records…"
  3. Success: "Syncing complete." The profile card switches to the real identity.

Merge path (account already exists)

  1. Steps 1–2 as above; linking detects the credential already belongs to an account.
  2. Confirm dialog: "This Google account already has SquawkIt data. Sync this device's records into it? You may see duplicates if you logged the same aircraft in both places."
  3. On Merge: progress ("Syncing your records…") → success ("Syncing complete.").
  4. On Cancel: return to guest, no account switch, data untouched.
Peak-end design The most anxious moment (committing an account while holding irreplaceable local data) is wrapped in a visible progress state and a calm, definite close. The experience ends on "Syncing complete," not a blank screen, so the user knows their records are safe.

7. Edge Cases & Risks

CaseExpected behavior
User cancels the provider sheetStay a guest; nothing changes.
Offline during upgradeLinking needs connectivity; show a clear "needs internet" message and let them retry. Data is never touched.
Link succeeds but cloud push is slow/offline afterwardUpgrade is considered done (account linked); syncing completes in the background, with sync status visible in Backup & Sync settings.
Chosen account already has data (collision)Merge path (R4).
Same aircraft logged in both guest and existing accountKnown v1 limitation: both copies appear (additive union). User can delete duplicates manually. Auto-dedupe is a future improvement.
App killed mid-mergeResumable, idempotent reconciliation on next launch (R8).
Apple "Hide My Email"Fully supported; the relay address is used as the account email.
Primary risk: merge duplicates Merging is an additive union of two record sets, not a semantic merge. A user who logged the same tail number in both places will see it twice. We accept this for v1 (clearly messaged), monitor the duplicate rate, and treat semantic dedupe as a fast-follow if the rate is high.

8. Scope & Phasing

PhaseIncludes
v1Feature Lab flag gate, Settings entry, Google (Android) + Apple (iOS) linking, clean-link sync, merge-on-collision, cancel/error safety, interruption recovery, "Syncing complete" success state.
Fast-followDuplicate detection/dedupe after merge; additional entry points (dashboard banner, contextual nudges) if conversion is low.
LaterEmail/password provider; account management (link/unlink multiple providers); guest-data import across reinstalls.
