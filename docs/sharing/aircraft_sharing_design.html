Aircraft Sharing — Design

How multiple accounts read and write the same aircraft on top of the R1 local-first store: access control, invites without email identity, sync-engine changes, and the technician-profile mirror. Companion to the PRD.

1. Starting Point & Constraints

Everything this design touches is shaped by four facts about the shipped system (see R1 design):

  1. All data is keyed by the owning user's UID, identically in both stores. Firestore: users/{uid}/aircraft/{acId} for the aircraft doc and users/{uid}/aircraft/{acId}/{maintenance_log|maintenance_task|maintenance_overview|squawk}/{id} for nested kinds. Locally, the SQLDelight entity table is keyed by (collection, scope_path, id) where scope_path is the same path (EntityScope). FirestoreRefs maps mechanically between the two.
  2. The sync engine is the only Firestore client. Feature managers read/write the local EntityStore only. Sync is per (CollectionKind, EntityScope): hydrate once, then pull-listen and push dirty rows. PushWorker.run(uid) currently filters the dirty queue to the /users/{uid}/% prefix.
  3. Entity payloads are opaque to the backend. Each synced doc is a SyncDocWire: Base64 proto bytes + deleted + schema + server timestamp. Security rules cannot inspect fields inside a payload — any data that rules must evaluate has to live in plain-field documents outside the entity path.
  4. Identity: Google / Apple / email-link / anonymous; anonymous users never sync. The self-technician convention already exists: UserInfo.self_technician_id points at the user's own Technician doc (see user-as-technician).

1.1 Data-placement decision

OptionSketchVerdict
A. Keep data under the hosting owner; grant access via an ACL Records stay at users/{hostUid}/aircraft/{acId}/…. A plain-field share document authorizes other UIDs; members' sync engines read/write the host's subtree directly. ✓ Chosen. Zero migration for existing data, zero change to entity payloads or LWW, one new top-level collection. The local store already supports foreign scope paths (scope_path is just a string).
B. Move shared aircraft to a top-level aircraft/{acId} collection Re-home docs on first share; all members point at the neutral location. ✕ Requires migrating every doc (and its subcollections) at share time and un-migrating at unshare, dual-path sync during transition, and rewriting local scope paths. Buys cleaner ownership-transfer later, but that's a non-goal; revisit if/when transfer ships.
C. Copy/fan-out records into each member's tree Cloud Function replicates writes N ways. ✕ N-way replication of a store whose whole design is single-source-of-truth + LWW; conflict resolution across copies is strictly worse. Rejected.

2. Data Model

2.1 The share tree (plain-field, rules-readable)

Field names are camelCase in the implementation The sketches below (and the rules in §4) use snake_case for readability, but the shipped documents use camelCase to match the client envelope and the functions schema — host_uid→hostUid, member_roles→memberRoles, display_name→displayName, technician_mirror→technicianMirror, writer_uid→writerUid, and so on. The canonical definitions live in backend/firebase/functions/src/sharing/sharingModels.ts and the deployed backend/firebase/firestore.rules.

One new top-level collection, keyed by aircraft id (aircraft ids are globally-unique random UUIDs, so no host prefix is needed):

Keyed by host, then aircraft (#204). An aircraft id is unique only within a user's tree. Keyed globally, the ACL had one slot per aircraft id — and anyone who knew an id could create that slot, honestly naming themselves hostUid, whereupon the rules (looking the ACL up by aircraft id alone) would consult their doc to authorize access to someone else's aircraft. Namespacing by host means the {hostUid} segment is pinned to request.auth.uid on create, so a share a caller can mint only ever governs the caller's own tree. The lookup side takes the host from the path of the resource being read (users/{ownerUid}/aircraft/{acId}) — never from a field, and never from anything the caller supplies.

aircraft_shares/{hostUid}/aircraft/{acId}    // root doc — the ACL, read by security rules
  host_uid:      string                     // account whose tree physically holds the data
  aircraft_id:   string                     // == {acId}, for collection-group queries
  member_roles:  map<uid, "owner"|"technician">   // includes host_uid → "owner"; denormalized for one-get rule checks
  created_at:    timestamp

aircraft_shares/{hostUid}/aircraft/{acId}/members/{uid}   // one per member — display + technician mirror
  role:          "owner" | "technician"
  display_name:  string                     // from Firebase Auth at redemption; member-updatable
  photo_url:     string?
  technician_mirror: {                      // §7 — maintained by the member's own client
    name, certificate_type, cert_number,
    cert_expiration, cert_expire_limit
  }?
  added_at:      timestamp
  invited_by:    uid

aircraft_shares/{hostUid}/aircraft/{acId}/invites/{tokenHash}  // pending invites; tokenHash = SHA-256(secret)
  role:          "owner" | "technician"
  created_by:    uid
  created_at:    timestamp
  expires_at:    timestamp                  // default now + 7d
  max_uses:      int                        // default 1
  use_count:     int
  revoked:       bool
  • These are plain-field docs, not SyncDocWire, because rules must read member_roles and functions must validate invites. They are deliberately outside the entity-sync path.
  • member_roles on the root doc is the single get() security rules perform per request (§4). The members subcollection carries everything else so the root doc stays tiny and low-contention. The redeem/revoke Cloud Functions keep the two in step transactionally.
  • The root doc is created lazily by the owner's client on first invite. Spoof-proofing: rules only allow create when host_uid == request.auth.uid and exists(/users/$(request.auth.uid)/aircraft/$(acId)) — you can only open a share for an aircraft that lives in your own tree.

2.2 SharedAircraftRef — the member-side pointer (synced kind)

A member's sync engine needs to know, at sign-in on any device, which foreign scopes to hydrate. That index lives in the member's own tree as a new synced entity kind, so it rides the existing hydrate/pull/push machinery and is available offline:

// core/model/src/commonMain/proto/sharing/shared_aircraft_ref.proto
syntax = "proto3";
option java_package = "dev.fanfly.wingslog.core.model.sharing";
option java_multiple_files = true;

message SharedAircraftRef {
  string aircraft_id = 1;
  string host_uid    = 2;
  ShareRole role     = 3;   // mirror of ACL role; UI hint only — enforcement is the ACL
}

enum ShareRole {
  SHARE_ROLE_UNKNOWN    = 0;
  SHARE_ROLE_OWNER      = 1;
  SHARE_ROLE_TECHNICIAN = 2;
}
  • New CollectionKind.SharedAircraftRef (wireName = "shared_aircraft_ref", schema sharing.SharedAircraftRef), registered in EntityCodecRegistry; a 9th kind. Scope: userRoot(uid), doc id = aircraft_id. Added to SyncEngine.TOP_LEVEL_KINDS.
  • Firestore path: users/{memberUid}/shared_aircraft_ref/{acId}, standard SyncDocWire format.
  • Written by the Cloud Functions, not the member's client (created on redeem, tombstoned on revoke/leave/delete — §3). SharedAircraftRef is three scalar fields, so the TypeScript functions encode the proto with protobufjs against the checked-in .proto; a tombstone (deleted: true) doesn't even need a payload. Function-written docs are indistinguishable from client-pushed ones to the sync engine.
  • The role field is advisory (fast UI gating without a network read); the ACL is authoritative and rules never consult the ref.

2.3 Local storage: no schema change

A member's device stores shared rows in the existing entity table under the host's scope path, e.g. scope_path = "/users/{hostUid}/aircraft/{acId}/" for the plane's logs. The primary key (collection, scope_path, id) already isolates them from the member's own data. Purge-on-revocation is one SQL delete by scope prefix. No new tables, no migrations.

3. Invite & Membership Lifecycle

3.1 Creating an invite (client-side, owner)

  1. Owner's client generates a 128-bit random secret, computes tokenHash = SHA-256(secret).
  2. Ensures aircraft_shares/{hostUid}/aircraft/{acId} exists (lazy create, §2.1), then writes invites/{tokenHash} with role/expiry/max_uses. Rules allow this for owners only.
  3. Renders the share URL — the web-app host, so a device without the app still lands on something that works:
    https://squawkit.web.app/share#{acId}.{base64url(secret)}
    The token rides in the URL fragment, which browsers do not send to servers — it can't leak into hosting/CDN logs. Android App Links / iOS Universal Links route the same URL into the native apps. The QR code encodes the same URL.

Only the hash is stored, so a read of the invites subcollection (or a backend leak) yields nothing redeemable; the secret exists only inside the link the owner chose to share. Cancelling an invite sets revoked: true.

3.2 Redemption — callable function redeemAircraftShareInvite

Redemption must write into two account trees (the share's member list and the member's own shared_aircraft_ref) and enforce invite validity atomically — a job for a callable Cloud Function with the Admin SDK (precedent: requestExportDelivery in backend/firebase/functions), not for client writes + rules:

redeemAircraftShareInvite({ aircraftId, secret })   // caller: any signed-in non-anonymous user
  1. require auth != null && !auth.token.firebase.sign_in_provider == "anonymous"
  2. tokenHash = sha256(secret); tx.read aircraft_shares/{hostUid}/aircraft/{acId} + invites/{tokenHash}
  3. validate: exists, !revoked, now < expires_at, use_count < max_uses
  4. if caller already in member_roles → return { alreadyMember: true } (token NOT consumed)
  5. tx.write:
       invites/{tokenHash}.use_count += 1
       member_roles[callerUid] = invite.role
       members/{callerUid} = { role, display_name, photo_url, added_at, invited_by }
       users/{callerUid}/shared_aircraft_ref/{acId} = SyncDocWire{ payload: SharedAircraftRef, … }
  6. return { aircraftId, hostUid, role, registration? }

The client then nudges SyncEngine.resyncCurrentUser(); the new ref doc arrives via the existing shared_aircraft_ref pull listener (or the resync's hydration), and the engine spins up the shared scopes (§5). Signed-out/guest redeemers park {acId, secret} in memory/prefs, complete sign-in or the guest upgrade, then call the function.

3.3 Revoke, leave, role change, delete

The tombstone is what tells the ex-member's devices to purge (§5.4). Hosting owner cannot be revoked (function-enforced).
LeaveThe memberSame function, memberUid = caller. Hosting owner cannot leave.

A Firestore trigger on users/{uid}/aircraft/{acId} (deleted == true) tears down aircraft_shares/{hostUid}/aircraft/{acId} and tombstones every member's ref. Members' aircraft rows also die naturally via the normal pull of the aircraft tombstone.
Division of labor Clients write only within their own rights (invite docs, their own member-doc mirror fields, entity data). Every operation that must touch two trees or enforce invite semantics is a Cloud Function transaction. Rules are the backstop; functions are the only cross-tree writers.

4. Security Rules

Sketch (paths abbreviated; role() does the single ACL get()):

function role(acId) {
  return get(/databases/$(db)/documents/aircraft_shares/$(acId))
           .data.member_roles[request.auth.uid];        // null/absent → no access
}
function isOwner(acId)  { return role(acId) == "owner"; }
function isMember(acId) { return role(acId) in ["owner", "technician"]; }

match /users/{uid} {
  // Own tree: unchanged — full access to your own data.
  match /{document=**} { allow read, write: if request.auth.uid == uid; }

  // Shared aircraft doc: members may GET (not LIST) it; only owners may write.
  match /aircraft/{acId} {
    allow get:   if request.auth.uid == uid || isMember(acId);
    allow write: if request.auth.uid == uid || isOwner(acId);

    // Nested maintenance data: any member may read and write.
    match /{kind}/{docId} {
      allow read, write: if request.auth.uid == uid || isMember(acId);
    }
  }
}

match /aircraft_shares/{acId} {
  allow get: if isMember(acId) ||
                exists(/…/users/$(request.auth.uid)/aircraft/$(acId));
  allow create: if request.resource.data.host_uid == request.auth.uid
                && exists(/…/users/$(request.auth.uid)/aircraft/$(acId));
  allow update, delete: if false;                        // functions only (member_roles is FN-managed)

  match /members/{memberUid} {
    allow read: if isMember(acId);
    // Member maintains own display + technician mirror; role is immutable from clients.
    allow update: if request.auth.uid == memberUid
                  && request.resource.data.role == resource.data.role;
    allow create, delete: if false;                      // functions only
  }
  match /invites/{tokenHash} {
    allow read, create: if isOwner(acId);
    allow update: if isOwner(acId);                      // cancel (revoked=true)
    allow delete: if false;
  }
}
Path (under users/{host}/aircraft/{acId})HostOwner memberTechnician member
aircraft doc itselfR/WR/W (get only, no list)R (get only)
maintenance_log, maintenance_task, maintenance_overview, squawkR/WR/WR/W
aircraft_shares/{hostUid}/aircraft/{acId} root + membersR; writes via functions/owner-scoped invite opsR (list visibility), no invite access
Known limitations, accepted for v1 (1) Record-type granularity only. Payloads are opaque proto bytes, so rules can't check fields — a technician who can write logs can write any bytes into a log doc. Matches the PRD's coarse permission matrix. (Authorship is still attested out-of-payload: §7.5 stamps a rules-enforced writer_uid on every write, so who wrote each revision is tamper-evident even though what they wrote isn't inspectable.) (2) Members can only get the aircraft doc, not list the host's aircraft collection — this is what keeps the host's other aircraft private, and it forces a doc-level (not collection-level) sync path for the shared aircraft doc (§5.2). (3) Each shared-scope request costs one extra rules get() — billed as a read, cached per request; acceptable at GA share sizes. (4) Technicians can read the aircraft doc but not the host's technician collection or anything else user-level — privacy boundary is the aircraft subtree.

5. Sync Engine Changes

5.1 Scope fan-out

Today SyncEngine.runForUser hydrates TOP_LEVEL_KINDS at the user root, then watches the local aircraft list to spin per-aircraft scopes. Sharing adds a parallel branch driven by the refs store:

runForUser(user):
  … existing own-tree branches …
  + hydrateAndListen(SharedAircraftRef, userRoot(uid))          // 9th top-level kind

  + observe EntityStore<SharedAircraftRef>.observeAll(userRoot(uid))
      .distinctUntilChanged { set of (hostUid, acId) }
      .collectLatest { refs →
        for (ref in refs) {
          watchSharedAircraftDoc(ref)                            // §5.2 — doc-level
          for (kind in PER_AIRCRAFT_KINDS)
            hydrateAndListen(kind, EntityScope.aircraftChild(ref.hostUid, ref.acId))
        }
      }

EntityScope.aircraftChild(hostUid, acId) already expresses foreign scopes — it's just segments. FirestoreRefs needs no change. The per-aircraft nested kinds use the existing collection listeners, which are legal for members under §4 rules.

5.2 Doc-level pull for the shared aircraft doc

The shared aircraft doc cannot be pulled with the existing collection-query listener: a member may get users/{host}/aircraft/{acId} but a query over users/{host}/aircraft is a list and is denied. Add a small FirestoreDocPullSubscription — a snapshot listener on a single DocumentReference — feeding the same PullListener.apply(remote) path (LWW, tombstones, cursor advance all unchanged). The row lands in the local store as kind Aircraft at scope userRoot(hostUid), exactly where FirestoreRefs would put it.

5.3 Push: widen the dirty-queue filter

PushWorker.run(uid) filters dirty rows to /users/{uid}/% (account isolation for signed-out queues). A member's edits to a shared plane are dirty rows under /users/{hostUid}/… and would never drain. Fix: the worker's prefix set becomes {"/users/{uid}/%"} ∪ {"/users/{ref.hostUid}/aircraft/{ref.acId}/%" for each live ref}, recomputed from the refs store (the observing flow re-collects when refs change). Ordering, batching, and LWW are untouched — a member's push writes to the host's docs, Firestore stamps the server time, every other member's listener picks it up. Multi-user conflicts resolve exactly like today's multi-device conflicts.

5.4 Revocation teardown & purge

The revoke function's ref tombstone is the purge signal — it arrives on the ex-member's devices through the ordinary shared_aircraft_ref pull listener, works for every device of the member, and needs no extra channel:

  1. Ref tombstone applies → the refs flow emits without that aircraft → collectLatest cancels its listeners (already the pattern for the own-aircraft branch).
  2. A SharedScopeJanitor compares live refs against local rows and deletes (hard SQL delete, not entity tombstones — these were never ours to tombstone) all rows and cursors with scope prefix /users/{hostUid}/aircraft/{acId}/, plus the aircraft row itself. Runs on refs-change and at engine start.
  3. Queued dirty rows in the purged scope are dropped; if any existed, surface the "changes discarded" notice from the PRD.
  4. Belt-and-braces: if revocation raced the listener (member offline at tombstone delivery but listeners resumed first), shared-scope pulls/pushes fail PERMISSION_DENIED. The engine treats a permission failure on a shared scope as "revoked": tear down + purge locally (the ref row is stale; overwrite it with a local tombstone). This also covers a deleted share doc.

Both directions of the race are handled (#144). A denied read reconciles from the shared watcher; a denied push — the member edited the aircraft offline, was revoked meanwhile, and the queued write lands before the tombstone does — reconciles from PushWorker, which hands the (hostUid, aircraftId) back to the engine and takes the same path. Without this, a denied push classified as AuthExpired and told the member their session had expired, while the rows stayed dirty and retried forever. Crucially the denial is scoped: PERMISSION_DENIED in the member's own tree still raises the auth banner, because there it means an expired token or a rules regression, not a revoke.

Because the engine now swallows the shared-scope denial, it must be counted instead — see the SyncTelemetry safety valves (PRD §9): sync_permission_denied_write (scope=own|shared) and sync_share_reconciled (trigger=denied_read|denied_write). A denial paired with a reconcile is this expected race; a denial without one is a UI gating bug, and would otherwise be invisible.

5.5 Gating

Shared-scope sync obeys the same master switches as everything else: signed-in, non-anonymous, cloudSyncEnabled. When sync is off, shared data is a frozen local copy; sharing management UI (which is online-only, §6.2) is disabled with explanatory copy.

6.2 SharingManager — a deliberate exception to "managers never touch Firestore"

Share ACL docs are plain-field, rules-inspected, function-written — they can't ride the entity-sync path. SharingManager therefore talks to Firestore/Functions directly, an online-only surface, following the existing precedent of ExportHistoryRemoteRepository/ExportDeliveryBackend in feature/export. Sharing management inherently requires connectivity (you're granting cloud access); the shared content remains fully local-first.

interface SharingManager {
  fun observeShareState(acId: String): Flow<AircraftShareState>   // members + invites (Firestore snapshots)
  fun observeMyRole(acId: String): Flow<ShareRole>                // local: refs store / own aircraft ⇒ instant, offline-correct

7. Technician Identity: the Profile Mirror

Requirement: a shared member who performs work must appear in the aircraft's technician machinery backed by their own license data — never typed by the aircraft owner.

Role and technician identity are orthogonal The share role answers "what may this account modify"; the mirror answers "how does this person appear on a signed log". Every member publishes a mirror regardless of role. An owner who is also a mechanic needs no dual role — Owner already supersets Technician on permissions — they just need their mirror published so other members can select them. Certificate fields may be empty (e.g. an owner doing FAR 43 preventive maintenance with no A&P certificate); a name-only linked entry is still the correct picker content.

7.1 Why a mirror (and who writes it)

The member's self-technician record lives at users/{memberUid}/technician/{selfId} — private, opaque proto, and its "self-ness" is only known via that user's UserInfo. Letting other members read it would require exposing part of a user's private tree and decoding payloads they shouldn't see in bulk. Instead, the member's own client publishes a small plain-field mirror into the share: aircraft_shares/{acId}/members/{uid}.technician_mirror (shape in §2.1). Rules already allow exactly this write (member updates own member doc). No proto decoding server-side, no privacy hole — the member is knowingly sharing precisely the fields that go on a signed log, per share.

7.2 Mirror lifecycle

  • On redemption: the redeem confirmation runs publishTechnicianMirror() — reads TechnicianManager.observeSelf() locally, writes the mirror to every share the user belongs to (here, the new one). Runs for every redeemer, Owner or Technician alike. The hosting owner publishes theirs when creating the share's first invite (lazy-create path, §3.1) — the one member who never redeems.
  • On profile edit: EditTechnicianScreen save path for the self-record triggers publishTechnicianMirror() for all of the user's memberships (enumerated from the local refs store; queued/retried while offline via a lightweight outbox in SharingManager). Freshness is best-effort; log snapshots always capture whatever is current at signing time — same semantics as today's picker.
  • On revocation/leave: member doc (and mirror) is deleted by the function. History inside logs is untouched — snapshots, as always.

7.3 Surfacing in the technician features

  • Picker (log form): for a given aircraft, TechnicianPickerSheet shows — (1) the current user's self record first (unchanged); (2) linked technicians: every other share member with a published mirror — membership-with-mirror, not role, is the listing criterion, so Owner-mechanics appear too — sourced from SharingManager.observeShareState(acId), marked with a "Linked" badge; (3) the aircraft-relevant manual entries as today. Selecting a linked technician snapshots the mirror fields into the log's embedded Technician proto — the snapshot mechanism is untouched, only the source differs. The embedded proto gains an optional source_uid — the owning uid of the picked first-party profile (the member's own self-record or another member's mirror; empty for manual entries) — so a log can show provenance ("from linked profile") and drive the self-signed check (§7.5).
  • Manage Technicians (Settings): stays a personal list, but linked technicians from shares the user belongs to are shown in a separate read-only section ("Linked from shared aircraft"), no edit or delete — tapping explains the data is maintained by that person. The existing manual CRUD is unchanged for off-app mechanics.
  • A shared technician logging their own work needs no mirror at all — their device has their real self-record locally; the picker defaults to it and snapshots it directly.

7.4 Reconciling manual entries with linked members

A user who hand-typed mechanics into Manage Technicians before sharing will, once those people join as members, see each of them twice: the stale manual row (user-level, in their own technician collection) and the live linked mirror (per-aircraft, in the share). The picker must collapse these without rewriting the immutable log snapshots that already captured the manual copy.

Matching & precedence

PairMatch keyResolution
manual + manualequal non-empty cert_numberAuto-merge candidate: keep the most-complete / most-recently-edited row, tombstone the other. Safe — logs hold snapshots.
manual + manual, name-onlynormalized name (+ cert_type)Proposed, never auto-applied — names collide; requires explicit confirmation.
manual + linked mirrorequal non-empty cert_number, else nameMirror is the source of truth. The manual row is aliased, not deleted (see below).
mirror + mirror—Never merged — two members are two distinct people/UIDs. Equal cert_number across mirrors is surfaced as a warning (likely a typo), not a merge.
Empty cert_number is never a match key The marquee case — an owner doing FAR 43 preventive maintenance with no certificate — is a name-only entry. Matching two blank certificate numbers would collapse every name-less technician into one. Name-only matches are always confirmation-gated.

Alias, not delete (the scope mismatch)

Manual technicians are user-global (they show for every aircraft); mirrors are per-aircraft (a member may be linked on one of the user's shared aircraft but not another). Hard-deleting a manual row because it matched a mirror on aircraft X would make that person vanish from the picker on aircraft Y. So the manual↔mirror merge is a soft alias: the Technician proto gains an optional superseded_by_uid. Where that member's mirror is present for an aircraft, the picker hides the aliased manual row and shows the mirror; on aircraft with no mirror for that person, the manual row still appears (labeled "not linked here"). superseded_by_uid rides the normal sync path, so the alias is consistent across the user's devices. The field is meaningful only on roster rows; it is ignored in the embedded log snapshot.

UI

A dismissible promo on TechnicianListScreen ("some technicians look like the same person — review") opens a review sheet: auto-safe groups pre-checked, ambiguous groups (name-only, multi-mirror) require a pick. Nothing is applied silently — a merge tombstones or aliases roster entries. A persisted "reviewed" flag prevents nagging. No log snapshot is ever touched — merging changes only the go-forward picker roster.

7.5 Authorship attestation (who wrote each record)

Because the log payload is opaque proto bytes and every member may write any bytes, a member can create a log and stamp a different member as its technician. Snapshot data alone can't distinguish "B signed their own work" from "A attributed work to B." The fix is to attest authorship outside the opaque payload, where rules can enforce it — the same move the ACL uses to be rules-readable.

  • Envelope field. SyncDocWire gains a plain writer_uid field (alongside deleted / schema / server timestamp — outside the Base64 payload). PushWorker stamps it with the pushing account's uid on every write.
  • Rule enforcement. On the shared aircraft subtree, allow write additionally requires request.resource.data.writer_uid == request.auth.uid. A client cannot forge request.auth.uid, so writer_uid is a tamper-evident, backend-attested record of who wrote each revision. No Cloud Function needed — rules alone enforce it.
  • Self-signed indicator. The log surfaces "signed" vs "assigned" by comparing the revision's writer_uid against the embedded snapshot's source_uid (§7.3): equal ⇒ "Logged and signed by {name}"; different or absent ⇒ "Logged by {author} · technician {name} (assigned)". LWW, tombstones, cursors, and the pull path are unchanged — writer_uid is just another envelope field carried through.
Limitation & the countersign fast-follow writer_uid reflects the latest revision's author, and attestation proves who wrote the row, not who performed the work. A durable per-log signature that survives later third-party edits — the named technician confirming from their own account before the entry counts as signed — is the countersign fast-follow (deferred with the Viewer role). v1 makes third-party attribution visible and unforgeable; countersign makes it a true signature.

8. Offline & Conflict Semantics (unchanged by design)

  • All members get full offline read/write on shared maintenance data; writes queue as dirty rows and drain via §5.3. Conflicts between members are the same whole-record LWW-on-server-timestamp as multi-device today; dirty rows remain immune to remote overwrite until pushed.
  • The only sharing-specific data-loss window is unsynced edits at revocation (PRD §8) — accepted and surfaced in UX.
  • Share ACL state (Manage Access) is online-only; role for UI gating comes from the offline-correct local refs store.

9. Attachments across shared aircraft

Implementation Status Designed here; not yet built. Sharing v1 ships an honest degradation (issue #146): a member sees the attachment exists and is told it is not available on a shared aircraft, the attach affordance is hidden, and plain link URLs in the payload still work. This section is the design for the fast-follow, tracked as epic #196 [P8] and its sub-issues (P8.1–P8.8). It supersedes the earlier "custom-claims, maybe" sketch.

Blob objects live in Firebase Storage under user-scoped paths, with matching Storage rules and a per-scope transfer pipeline keyed to /users/{uid}/% (see R2 design). An attachment on aircraft acId hosted by hostUid lands at users/{hostUid}/aircraft/{acId}/blobs/{blobId}. Firestore record data crosses the account boundary via the aircraft_shares ACL; the blobs hanging off those records do not.

9.1 The blocker is structural

storage.rules is strictly request.auth.uid == userId for read and write, so a member can neither fetch a host's attachment nor upload one. This is not a missing if: Storage rules cannot get() Firestore, so they cannot consult the aircraft_shares ACL to authorize a cross-tree read. Crossing the boundary needs either a server-side broker or custom claims.

9.2 Chosen approach — signed-URL broker

A Cloud Function is the only path across trees. A callable checks the aircraft_shares ACL and mints short-lived URLs; storage.rules stay uid-scoped and closed.

ApproachHowVerdict
A. Signed-URL broker Callable verifies membership in aircraft_shares/{acId}, then returns a short-lived signed download URL (reads) or a resumable-upload session URL (writes) into the host's tree. Rules never change. Chosen. Revocation cuts off new URLs at once — a revoked member stops being issued any; already-minted URLs lapse within the short TTL (§9.2.1). Cost: one function hop per open/upload.
B. Custom claims Put shared aircraft ids in the auth token so Storage rules check them directly. Rejected (same as sharing v1): claims have a size cap, and revocation does not take effect until the token refreshes — an ex-member keeps blob access for up to an hour. A security regression relative to Firestore, where revocation is immediate.

The broker lives under backend/firebase/functions/src/storage/ (alongside blobRefs.ts, storageSweep.ts) and reuses the ACL-check helper the existing sharing functions already use (revokeAircraftShare, updateAircraftShareRole, onAircraftDeleted).

9.2.1 Broker specifics

A signed URL is a bearer credential: the ACL is checked once, at mint time, and the URL then works for anyone who holds it until it expires. That shapes four answers (raised on #243).

ConcernAnswer
How short is "short-lived"? Downloads: 5 minutes. Long enough to fetch one blob on a slow link, short enough to bound every risk below. The client fetches immediately and does not persist the URL. Uploads: the resumable session is minted per-upload and consumed at once; it is bounded to the single object it was issued for, not a standing grant.
Revoked right after the URL is minted? The already-minted URL keeps working until it expires — a residual window of ≤ the TTL (5 min). This is the one place revocation is not instantaneous, and it is a deliberate, bounded trade: still an order of magnitude tighter than custom claims (up to an hour, §9.2) and far below Firestore's own revocation, which is immediate. A just-revoked member can, at worst, finish reading blobs they could already see, for up to five minutes. If we ever need hard-zero, the escalation is a streaming proxy (below).
URL leaked to someone without access? Same bearer property: whoever holds it can read that one blob for ≤ 5 min, then it is dead. Mitigations: short TTL; the URL is scoped to a single object, never a folder or a token that unlocks the tree; HTTPS only; the broker never logs full signed URLs. It is not a login and grants nothing beyond that one file for that short window.
Need hard-zero revocation / leak-resistance? Escalation, not v1: a streaming proxy — the function itself reads the object and pipes the bytes, re-checking the ACL on every request, so nothing bearer is ever handed out and revocation is instant. Cost: the function pays egress and holds the connection for the whole transfer, which is why v1 prefers signed URLs. Flagged for you as the lever to pull if the 5-minute window is unacceptable.
Does a slow download fail at the TTL? No — the TTL gates when a request may begin, not how long an accepted transfer runs. GCS validates the signed URL's expiry when it authorizes the start of the GET (same as S3 presigned URLs); once the response body is streaming it runs to completion, so a fetch that starts at 4:59 and takes another ten minutes still succeeds. The only thing the TTL kills is re-initiation: a dropped-and-retried connection, or a ranged/resumable read that opens a fresh request after expiry. BlobDownloadDriver is a one-shot GET that, on transient failure, re-fetches through the broker — so a retry simply mints a new URL (a cheap function hop), never a stuck download. Two things keep this off the hot path entirely: attachments are MB-scale (a few MB downloads well inside 5 min even on a poor link), and uploads don't use this mechanism at all — resumable upload sessions are built for long, interruptible transfers (ACL checked once at session creation, ~week-long lifetime). If we ever host large blobs on slow links, the download TTL is just a knob: raise it and the residual-revocation window of §9.2.1 grows to match.

How many functions, and what are they? The broker itself is two callables, kept separate because their inputs and return shapes differ:

  • getBlobDownloadUrl(hostUid, acId, blobId) → ACL check → short-lived signed GET URL.
  • getBlobUploadSession(hostUid, acId, blobId) → ACL + entitlement check → resumable upload session URL, stamping writer_uid.

Two more functions live in P8 but are not the broker: the entitlement projector that maintains aircraft_shares.attachmentsEnabled on subscription change (§9.7, P8.7), and the existing storageSweep that reclaims unreferenced/tombstoned bytes (§9.6, already shipped in P7). So: two new broker callables, one new projector, and one reused sweep — four functions touch attachments-on-shared, of which the broker is two.

9.3 One canonical blob, no mirrors

Requirement 2: one canonical blob per attachment, in the host's tree. Copying blobs per member is explicitly out of scope — it multiplies storage cost and creates divergent copies of the same file. Every member reads and writes the same object through the broker; only local caches are per-device.

9.4 Client scope widening

Today the blob machinery only ever looks at the signed-in user's own tree, so a member's blobs on a shared aircraft are invisible to sync. Mirror the push-queue prefix widening of §5.3. Known gaps (all need fixing regardless of approach):

  • SqlDelightLocalBlobStore — the scheduler only scans "/users/{myUid}/%", so a member's shared-aircraft blobs are never uploaded or downloaded.
  • AircraftOverviewViewModel.observeBlobStates — hardcoded to /users/{myUid}/aircraft/{acId}/, the wrong prefix for a shared aircraft, so sync state never resolves.
  • BlobIndexReconciler prefetch and the upload/download/delete drivers assume own-tree paths.

AircraftScopeResolver already resolves an aircraft id to the correct host-scoped EntityScope (own vs. shared, via the member's shared_aircraft_ref); the widened scopes drive off it rather than re-deriving paths. The drivers then branch on scope: own tree → the existing direct Storage path; foreign host → the §9.2 broker.

9.5 Authorship

A member's upload is attributable, exactly as writer_uid attests record writes (§7.5): the broker stamps the uploading member on the object so "who added this" survives across accounts.

9.6 Revoke and deletion

  • Revoke never deletes host bytes (requirement 3). Removing member B purges B's local cache only; the canonical blob stays put and reachable for the host and every remaining member. revokeAircraftShare / onAircraftDeleted must not touch users/{hostUid}/…/blobs.
  • Deletion cleans up for everyone (requirement 4). Deleting an attachment, its parent record, or the aircraft tombstones the blob, and the cross-account sweep from [P7] #155 (onRecordDeleted.ts, storageSweep.ts) reclaims the bytes — no orphans accruing cost. A member-initiated delete must produce a tombstone the sweep picks up, not be dropped as a foreign-tree write.

9.7 Entitlement — the storage owner pays

Attachments are gated (behind attachmentUploadEnabled in Feature Lab today, a paywall later), so the design takes an entitlement, not a dev flag. The decisive move: the gate is a property of the aircraft, not the user.

attachmentsEnabled(acId) = entitlement(hostUid of acId)

Whoever's storage holds the blob is who pays for it — the hosting owner. So an unsubscribed technician can view and upload (not view-only) on an aircraft whose owner has paid, and gets nothing on their own unpaid tree. A member must not be able to read the host's billing, so the host's entitlement is projected onto a rules-readable boolean on the ACL root — the same move the ACL already makes for roles: the fact is readable, the payload stays private.

aircraft_shares/{acId}: { hostUid, memberRoles, attachmentsEnabled: true }

A function maintains attachmentsEnabled when the host's subscription changes; the broker (§9.2) and the client drivers (§9.4) consult it. The owner-only toggle surfaces inside the Edit Aircraft page, where the aircraft's other properties live — making the scoping legible: you are enabling attachments for this aircraft, on the owner's storage and the owner's bill.

Decisions Lapse policy: when a host's subscription expires mid-share, existing attachments stay readable and only uploads stop — making paid-for data unreadable the moment a card fails is hostile. Quota: no cap on what a technician uploads on the host's behalf — a mechanic mid-job must never be blocked by a limit; storage-weight visibility (not enforcement) is a separate general enhancement, #199, and is not blocked behind P8.

9.9 Sequence diagrams

Three flows a shared aircraft has to get right. Across all of them the record — and the Attachment{id, sha256, mime, size} it carries — already syncs cross-account through the P3 shared-scope fan-out, so discovery is free; only the bytes are gated, and the broker is the only door to them.

10. Testing

  • Security rules (highest-value): Firestore emulator suite in backend/firebase — member/non-member/technician matrix over every path in §4, get-vs-list on the aircraft doc, share-doc create spoofing, member-doc role immutability, invite access, and writer_uid attestation (a write whose writer_uid ≠ request.auth.uid is rejected — §7.5).
  • Functions: unit tests for redeem (expiry, reuse, revoked, already-member, anonymous caller), revoke (hosting-owner protection, tombstone emission), delete-trigger teardown.
  • Sync engine (existing MockK/Truth harness per kmm-test-writer): refs-driven scope spin-up/teardown; push-prefix widening drains foreign-scope dirty rows; doc-level pull applies LWW; janitor purges on ref tombstone; PERMISSION_DENIED on a shared scope triggers teardown+purge, not the failure banner.
  • Managers/UI: scope-resolver switching own vs. shared; fleet combine emits shared entries with badges; role gating hides owner-only actions; picker shows linked entries and snapshots mirror data; duplicate-merge precedence (manual↔manual collapse, manual↔mirror alias via superseded_by_uid, mirror↔mirror never merged, empty cert_number never a key); self-signed vs. assigned indicator derived from writer_uid vs. source_uid.
  • End-to-end (manual, two accounts): invite→QR→redeem→hydrate; concurrent edits; offline edit + revoke; aircraft delete propagation.

12. Open Questions

  1. Web QR scanning: web app renders QR + link fine; scanning is camera-API work. Proposal: v1 web redeems via link only.
  2. Display-name staleness in member docs (auth display name changes aren't mirrored automatically). Proposal: refresh own member docs opportunistically at app start, same outbox as the technician mirror.
  3. Should Owners auto-publish a technician mirror? — RESOLVED: yes, every member publishes one regardless of role. Role (permissions) and technician identity (appearing in the picker with license data) are orthogonal axes: Owner already supersets Technician on permissions, and owner-members who wrench on the plane (very common in partnerships, and the FAR 43 preventive-maintenance case) must be selectable by other members. The mirror is exactly the fields already stamped on a signed log, so publishing it adds no privacy exposure; certificate fields may simply be empty for non-certificated owners (name-only entry). See §7.
  4. Rules get() read cost per shared-doc operation: measure in dogfood; if material, revisit custom claims (claims add invalidation complexity — deliberately not chosen for v1).
  5. Hydration fan-out for members of many shared aircraft hits the same N+1 as own aircraft (R1 open question 5); same collection-group-query optimization applies later if telemetry warrants.
  6. Merge match precision (§7.4): name-only matching risks false positives (two real "J. Smith"s). v1 requires explicit confirmation for every name-only merge and never auto-applies; tune thresholds from dogfood.
